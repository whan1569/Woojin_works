# 특수기종 업체별 파생 코드 관리 가이드라인 보고서

작성자: 이동환
목적: 특수기종 및 업체별 파생 코드의 추적성 확보와 유지보수성 개선

---

## 1. 개요

본 보고서는 특수기종 및 업체별 요구사항으로 인해 중앙 프로젝트에서 파생되는 코드의 관리 방식을 개선하기 위한 가이드라인을 정리한 문서이다.

기존에는 여러 특수기종이 중앙 프로젝트를 기준으로 복사되어 각 업체 사양에 맞게 수정되는 방식으로 개발이 진행되었다. 그러나 시간이 지나면서 업체별 코드 갈래가 많아지고, 담당자마다 관리 방식이 달라지면서 수정 이력과 적용 기준을 추적하기 어려운 문제가 발생하였다.

특히 기존 코드를 기반으로 새로운 구조를 추가하기보다는, 기존 파일 내부 로직을 업체 요구사항에 맞게 변경하는 경우가 많았다. 이 경우 변경된 코드가 명확히 관리되지 않으면, 담당자가 퇴사하거나 시간이 지난 뒤 해당 코드가 어떤 업체 사양을 반영한 것인지 파악하기 어렵다.

이에 따라 본 가이드라인에서는 업체별로 변경된 코드를 별도 파일로 객체화하고, SVN에 관리 대상으로 등록하는 방식을 제안한다.

---

## 2. 기존 관리 방식의 문제점

### 2.1 업체별 파생 코드의 추적 어려움

특수기종은 중앙 프로젝트를 기반으로 하되, 업체별 요구사항에 따라 일부 로직이 변경되는 경우가 많다.

그러나 기존 방식에서는 변경된 코드가 명확한 명명 규칙 없이 관리되거나, 기존 파일을 직접 덮어쓰는 방식으로 처리되는 경우가 있었다. 이로 인해 시간이 지나면 다음과 같은 문제가 발생한다.

* 어떤 업체를 위해 변경된 코드인지 확인하기 어려움
* 원본 코드와 변경 코드의 차이를 추적하기 어려움
* 담당자 퇴사 시 변경 사유와 적용 기준이 사라짐
* 유사 특수기종 제작 시 기존 변경 코드를 재활용하기 어려움
* 동일하거나 유사한 로직이 여러 프로젝트에 중복 생성됨

---

### 2.2 담당자 의존적인 코드 관리

업체별 특수기종 대응은 현장 상황, 업체 요구사항, 옵션 구성 등에 따라 세부 로직이 달라질 수 있다.

기존에는 이러한 변경 사항이 담당자의 기억이나 개별 로컬 작업물에 의존하는 경우가 많았다. 이 경우 담당자가 변경되거나 퇴사하면 해당 코드가 어떤 목적과 기준으로 수정되었는지 파악하기 어려워진다.

결과적으로 이후 유지보수 담당자는 기존 코드를 다시 분석해야 하며, 경우에 따라 이미 해결된 문제를 반복해서 수정하게 되는 비효율이 발생한다.

---

### 2.3 중앙 프로젝트와 업체별 변경본의 경계 불명확

중앙 프로젝트는 여러 기종의 기준이 되는 공통 코드 역할을 해야 한다. 하지만 업체별 대응 과정에서 중앙 코드가 직접 수정되거나, 수정본이 명확히 분리되지 않으면 공통 기준과 업체별 변경 사항의 경계가 흐려진다.

이로 인해 다음과 같은 문제가 발생할 수 있다.

* 중앙 원본 코드가 어떤 상태였는지 확인하기 어려움
* 업체별 변경 사항이 공통 코드에 혼입될 가능성 증가
* 다른 업체 프로젝트에 불필요한 특수 로직이 전파될 가능성 증가
* 수정 영향 범위 검토가 어려워짐

따라서 중앙 코드와 업체별 변경 코드는 명확히 분리되어 관리될 필요가 있다.

---

## 3. 개선 방향

본 가이드라인의 핵심은 업체별로 변경된 코드를 명확한 파일명 규칙에 따라 별도 관리하고, 이를 SVN 관리 대상으로 등록하는 것이다.

즉, 기존 파일을 직접 덮어쓰는 것이 아니라, 다음과 같은 방식으로 업체별 변경본을 생성한다.

```text
기존 코드명 + 업체명
```

예를 들어 기존 파일이 다음과 같다고 가정한다.

```text
ht_org.c
```

이 파일을 Samsung 업체 사양에 맞게 수정했다면, 변경된 파일은 다음과 같이 관리한다.

```text
ht_org_samsung.c
```

이렇게 하면 파일명만 보아도 다음 정보를 확인할 수 있다.

* 기준이 된 원본 코드: `ht_org.c`
* 적용 대상 업체: `Samsung`
* 관리 목적: Samsung 사양에 맞게 변경된 파생 코드

---

## 4. 파일 관리 규칙

### 4.1 업체별 변경본 생성 규칙

업체별 요구사항으로 인해 기존 코드가 변경되는 경우, 변경본은 기존 파일명을 기준으로 업체명을 추가하여 생성한다.

예시:

```text
원본 파일:
ht_org.c

Samsung 업체 변경본:
ht_org_samsung.c
```

다른 업체에 대해서도 동일한 규칙을 적용한다.

```text
ht_org_lg.c
ht_org_hyundai.c
ht_org_woojin.c
```

이를 통해 동일한 기능 영역의 코드라도 업체별 변경 이력을 파일 단위로 구분할 수 있다.

---

### 4.2 SVN 등록 규칙

생성된 업체별 변경본은 반드시 SVN 관리 대상으로 등록한다.

업체별 변경본을 SVN에 등록하는 목적은 다음과 같다.

* 업체별 수정 코드의 이력 보존
* 담당자 변경 시 인수인계 가능
* 유사 특수기종 개발 시 기존 코드 재활용
* 원본 코드와 변경 코드의 비교 가능
* 코드 손실 및 로컬 의존성 방지

따라서 업체별로 수정된 파일은 개인 PC나 임시 폴더에만 보관하지 않고, 프로젝트 관리 범위 내에서 SVN에 업로드해야 한다.

---

## 5. 실제 적용 방식

업체별 변경본을 실제 프로젝트에 적용할 때는 기존 빌드 대상 파일명과의 관계를 명확히 관리해야 한다.

예를 들어 프로젝트에서 실제로 빌드되는 파일명이 `ht_org.c`라면, Samsung 업체용 변경본을 적용할 때 다음과 같이 처리한다.

```text
1. 기존 ht_org.c를 ht_org_static.c로 변경
2. ht_org_samsung.c를 ht_org.c로 변경
3. 변경된 ht_org.c를 기준으로 빌드 및 테스트 수행
```

적용 전:

```text
ht_org.c
ht_org_samsung.c
```

적용 후:

```text
ht_org_static.c
ht_org.c
```

여기서 `ht_org_static.c`는 기존 기준 파일을 보존하기 위한 용도이며, 새로 적용된 `ht_org.c`는 실제 프로젝트에서 사용되는 업체별 변경본이다.

---

## 6. 명명 규칙 예시

### 6.1 기본 규칙

```text
기존파일명_업체명.c
```

예시:

```text
ht_org.c
ht_org_samsung.c
```

### 6.2 적용 시 보존 파일명

```text
기존파일명_static.c
```

예시:

```text
ht_org_static.c
```

### 6.3 전체 예시

```text
원본 기준 파일:
ht_org.c

Samsung 업체 변경본:
ht_org_samsung.c

실제 Samsung 프로젝트 적용 시:
ht_org.c          → ht_org_static.c
ht_org_samsung.c → ht_org.c
```

## 7. 기대 효과

### 7.1 업체별 변경 이력 추적성 확보

업체별 변경 파일이 명확한 이름으로 SVN에 남기 때문에, 시간이 지난 뒤에도 어떤 업체를 위해 어떤 코드가 작성되었는지 추적할 수 있다.

이는 담당자가 퇴사하거나 프로젝트가 장기간 유지보수되는 상황에서도 중요한 기준 자료가 된다.

---

### 7.2 유지보수성 향상

기존 방식에서는 특정 업체 사양을 다시 확인하기 위해 여러 프로젝트를 직접 열어보고 코드를 비교해야 했다.

그러나 업체별 변경본이 명확히 관리되면, 유지보수 담당자는 해당 업체 파일을 기준으로 빠르게 변경 내용을 확인할 수 있다.

예를 들어 Samsung 관련 문제가 발생하면 `ht_org_samsung.c` 또는 이를 적용한 `ht_org.c`를 기준으로 분석할 수 있다.

---

### 7.3 코드 재사용성 향상

특수기종은 완전히 새로운 구조로 개발되기보다는, 기존 구조를 기반으로 일부 로직을 변경하는 경우가 많다.

따라서 업체별 변경본을 남겨두면, 향후 유사한 업체 또는 유사 옵션 대응 시 기존 코드를 참고하거나 재사용할 수 있다.

이는 신규 특수기종 대응 시간을 줄이고, 이미 검증된 로직을 활용할 수 있다는 장점이 있다.

---

### 7.4 중앙 프로젝트 오염 방지

업체별 변경 사항을 별도 파일로 관리하면, 중앙 프로젝트의 공통 기준 코드와 업체별 특수 로직을 분리할 수 있다.

이를 통해 특정 업체만을 위한 로직이 공통 코드에 섞이는 문제를 줄일 수 있으며, 중앙 프로젝트의 기준성을 유지할 수 있다.

---

## 8. 적용 시 주의사항

### 8.1 단순 백업 파일로 관리하지 말 것

업체별 변경본은 단순 백업 파일이 아니라, 업체별 특수 사양을 기록하는 관리 대상 파일이다.

따라서 임의의 파일명으로 저장하거나 개인 PC에만 보관해서는 안 된다.

잘못된 예:

```text
ht_org_new.c
ht_org_1.c
ht_org_test.c
ht_org_backup.c
```

올바른 예:

```text
ht_org_samsung.c
ht_org_lg.c
ht_org_hyundai.c
```

---

### 8.2 업체명 기준을 통일할 것

업체명은 가능한 한 일관된 표기 방식을 사용해야 한다.

예를 들어 Samsung을 다음과 같이 여러 방식으로 혼용하면 추적성이 떨어진다.

```text
ht_org_samsung.c
ht_org_sam.c
ht_org_ss.c
ht_org_sec.c
```

따라서 업체명 표기는 프로젝트 내에서 하나의 기준으로 통일해야 한다.

---

### 8.3 실제 적용 파일과 보존 파일을 구분할 것

실제 빌드 대상 파일은 기존 프로젝트 구조상 정해진 파일명을 사용해야 할 수 있다. 이 경우 업체별 변경본을 적용하더라도 기존 기준 파일을 삭제하지 않고 `_static` 명칭으로 보존한다.

예시:

```text
ht_org.c          → ht_org_static.c
ht_org_samsung.c → ht_org.c
```

이를 통해 실제 적용 파일과 기존 기준 파일을 동시에 확인할 수 있다.

---

## 9. 결론

특수기종 및 업체별 파생 프로젝트가 증가할수록 코드 관리의 핵심은 단순히 코드를 수정하는 것이 아니라, 어떤 업체를 위해 어떤 코드가 변경되었는지를 추적 가능하게 남기는 것이다.

본 가이드라인은 기존 중앙 프로젝트를 기반으로 업체별 변경본을 생성할 때, `기존코드명_업체명.c` 형식으로 파일을 별도 관리하고 SVN에 등록하는 방식을 제안한다.

또한 실제 프로젝트 적용 시 기존 기준 파일은 `_static` 명칭으로 보존하고, 업체별 변경본을 기존 빌드 대상 파일명으로 교체하여 적용한다.

이를 통해 업체별 특수기종 코드의 추적성, 재사용성, 유지보수성을 확보할 수 있으며, 담당자 변경이나 퇴사 이후에도 코드 관리 기준을 유지할 수 있다.

본 방식은 특수기종 개발 과정에서 발생하는 코드 분산 문제를 줄이고, 향후 유사 업체 대응 및 유지보수 작업의 효율성을 높이는 기준이 될 수 있다.
