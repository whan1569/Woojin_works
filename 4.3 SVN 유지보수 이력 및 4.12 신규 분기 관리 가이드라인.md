# 4.3 SVN 유지보수 이력 및 4.12 신규 분기 관리 가이드라인

## 1. 작성 목적

본 문서는 기존 4.3 버전 SVN의 유지보수 이력과 버전관리상 확인된 문제를 정리하고, 4.12 버전 신규 SVN 분기 생성 시 적용한 관리 범위와 향후 관리 기준을 정의하기 위해 작성한다.

기존 4.3 버전은 장기간 유지보수되면서 여러 기능 추가와 버그 수정이 이루어졌으나, 프로젝트 전체 구성 파일에 대한 버전관리 기준이 명확하지 않아 체크아웃 및 빌드 환경에서 반복적인 문제가 발생하였다.

이에 4.12 버전 신규 분기를 생성하면서 기존 문제를 정리하고, 향후 동일한 문제가 재발하지 않도록 관리 기준을 수립한다.

---

## 2. 기존 4.3 버전의 주요 문제

### 2.1 C 코드 중심의 부분 버전관리

기존 4.3 버전은 주로 C 코드 파일 수정 위주로 SVN에 커밋되는 방식으로 관리되었다.

그러나 Automation Studio 프로젝트는 C 코드만으로 구성되지 않는다.
정상적인 체크아웃과 빌드를 위해서는 다음 항목들이 함께 관리되어야 한다.

```text
- C 소스 코드
- 라이브러리 참조 파일
- 프로젝트 설정 파일
- 빌드 설정
- Logical / Physical 구성 파일
- 장비 옵션 및 환경 설정 파일
- 프로젝트 구동에 필요한 패키지 및 참조 파일
```

기존에는 이 중 일부 파일이 버전관리 대상에서 제외되거나 ignore 처리되어 있었고, 이로 인해 SVN에서 프로젝트를 새로 체크아웃해도 동일한 개발 환경이 재현되지 않는 문제가 발생하였다.

---

### 2.2 라이브러리 미인식 문제

기존 4.3 버전의 가장 큰 문제 중 하나는 체크아웃 시 라이브러리가 정상적으로 인식되지 않는 현상이다.

라이브러리 또는 참조 관련 파일 일부가 SVN 관리 밖에 있었기 때문에, 프로젝트를 새로 체크아웃한 후 작업자가 직접 라이브러리를 다시 추가하거나 환경을 수동으로 맞춰야 했다.

이 구조에서는 다음과 같은 문제가 발생한다.

```text
1. 작업자마다 프로젝트 환경이 달라진다.
2. 특정 PC에서는 빌드되지만 다른 PC에서는 빌드되지 않을 수 있다.
3. 신규 작업자가 프로젝트를 이어받을 때 환경 재구성이 필요하다.
4. 라이브러리 버전 차이로 인해 잠재적인 빌드 오류 또는 동작 차이가 발생할 수 있다.
5. SVN 이력만으로 실제 빌드 가능한 상태를 재현하기 어렵다.
```

결과적으로 기존 4.3 버전은 “SVN에 코드 이력은 존재하지만, 실제 빌드 가능한 프로젝트 상태는 작업자 로컬 환경에 의존하는 구조”가 되었다.

---

### 2.3 TEMP, 바이너리, Diagnosis 파일 혼재

기존 프로젝트에는 어느 시점부터 TEMP 파일, 바이너리 파일, Diagnosis 관련 파일 등이 포함되면서 관리 기준이 불명확해진 것으로 보인다.

해당 파일들이 정확히 어떤 시점에 어떤 이유로 포함되었는지는 명확하지 않지만, 결과적으로 다음과 같은 문제가 발생할 수 있다.

```text
- 자동 생성 파일과 필수 관리 파일의 구분이 어려워짐
- 불필요한 파일이 SVN에 포함되어 프로젝트 구조가 복잡해짐
- 반대로 필요한 파일이 ignore 처리되어 체크아웃 후 누락될 가능성 발생
- 파일 변경 이력에서 실제 수정 사항과 산출물 변경 사항이 혼재됨
- 프로젝트 용량 및 변경 추적 부담 증가
```

따라서 4.12 분기에서는 TEMP, 바이너리, Diagnosis 관련 파일에 대해 명확한 포함/제외 기준을 정해야 한다.

---

## 3. 4.3 버전 주요 유지보수 이력

입사 이후 기존 4.3 버전에서 수행한 주요 수정 이력은 다음과 같이 분류할 수 있다.

```text
1. 기능 추가 및 개선
2. 현장 버그 수정
3. 화면 및 UI 관련 수정
4. 장비 옵션 및 특수기능 대응
5. 프로젝트 구조 및 유지보수성 개선
```

주요 작업 항목은 다음과 같다.

```text
- 기종별 데이터 모델 관련 버그 수정
- 강제후퇴 시간 자동보정
- 게이트 옵션 UI 위치 자동 할당
- Machine Information / QR 관련 오류 수정
- 파일 관리 및 유지보수성 개선
- DATA492 관련 오류 수정
- 에러창 HTML 코드 인식 문제 해결
- 냉각모터 출력 안됨 문제 해결
- 로그 에러 입력단위 문제 해결
- Userlog 창 정보 출력 구조 리빌드
- 서보온도 2존 온도표기 안됨 해결
- 러너 관련 온도표시 버그 해결
- 태국어 리소스 추가
```

해당 작업들은 단순 기능 추가뿐 아니라, 실제 현장에서 발생한 오류를 수정하고 장비 운용 안정성을 높이는 방향으로 진행되었다.

## 4. 수정 작업 시 확인된 구조적 특징

### 4.1 옵션 기능과 특수기능의 복합성

사출기는 기본 구조는 유사하더라도 실제 현장에서는 특수기종과 옵션 구성이 다양하다.

따라서 특정 기능을 수정할 때 단순히 평균적인 기종 기준으로 판단하면 실제 장비에서 문제가 발생할 수 있다.

특히 다음 항목은 수정 시 반드시 연결 구조를 확인해야 한다.

```text
- 옵션 사용 여부
- 화면 표시 조건
- 내부 상태 변수
- 출력 조건
- 래더 접점
- 실제 DO 출력
- 안전 조건 및 인터록
```

옵션 기능은 단순히 기능 ON/OFF만 보는 것이 아니라, 화면, 내부 변수, 상태값, 최종 출력까지 이어지는 전체 흐름을 확인해야 한다.

---

### 4.2 출력 직접 제어의 한계

현장 대응 중 일부 출력 문제는 겉으로 보면 특정 DO를 직접 ON 시키면 해결될 것처럼 보인다.

예를 들어 냉각수 차단밸브 출력 문제의 경우, 단순히 다음과 같은 방식으로 접근할 수 있다.

```c
if (MA_STAT.AbWaterFlow_Use)
{
    DO_18_1 = 1;
}
```

그러나 실제 구조에서는 `DO_18_1`이 사이클 중간에서 직접 제어되는 값이 아니라, 별도의 내부 변수 또는 래더 접점을 통해 최종 출력되는 구조일 수 있다.

이 경우 사이클 중간에서 `DO_18_1`을 직접 ON 시켜도 이후 출력 처리 과정에서 다른 변수에 의해 다시 덮어써질 수 있다.

따라서 출력 문제를 수정할 때는 다음 흐름을 확인해야 한다.

```text
조건 변수
    ↓
내부 제어 변수
    ↓
래더 접점 또는 출력 매핑
    ↓
실제 DO 출력
```

즉, 수정 대상은 실제 DO 주소가 아니라 해당 DO를 최종적으로 구동하는 내부 변수일 수 있다.

---

### 4.3 판단 근거 기록 필요성

기존 작업 이력에는 버그 수정 이력은 남아 있으나, 수정 당시의 판단 근거가 충분히 남지 않는 경우가 있다.

장비 프로그램에서는 특정 수정이 왜 필요한지, 어느 변수와 연결되어 있는지, 어떤 영향 범위가 있는지를 남겨야 이후 유지보수가 쉬워진다.

따라서 향후 커밋 또는 수정 보고 시 다음 항목을 함께 기록하는 것이 바람직하다.

```text
- 발생 증상
- 원인 분석
- 수정 대상
- 수정 방식
- 영향 범위
- 테스트 결과
- 관련 옵션 또는 기종 조건
```

## 5. 4.12 신규 분기 생성 목적

4.12 신규 SVN 분기 생성은 단순히 Automation Studio 버전을 업그레이드하는 작업이 아니다.

기존 4.3 버전에서 누적된 버전관리 문제와 프로젝트 구조 문제를 정리하고, 향후 유지보수 가능한 기준점을 새로 만드는 작업이다.

4.12 신규 분기의 목적은 다음과 같다.

```text
1. 체크아웃 시 라이브러리 미인식 문제 제거
2. 프로젝트 의존성 및 참조 구조 정리
3. TEMP, 바이너리, Diagnosis 관련 파일 기준 정리
4. 기존 4.3에서 확인된 반복 오류 반영
5. 신규 작업자가 동일한 환경에서 바로 빌드 가능한 구조 확보
6. 작업자 로컬 환경 의존도 제거
7. 향후 유지보수 기준점 확보
```

## 6. 4.12 신규 분기 실제 관리 범위

4.12 신규 SVN 분기 생성 시, 프로젝트 체크아웃 후 동일한 개발 환경을 재현할 수 있도록 관리 대상과 제외 대상을 명확히 구분하였다.

기존 4.3 버전에서는 C 코드 위주로만 관리되고, 라이브러리 및 프로젝트 구성 파일 일부가 SVN 관리 밖에 있어 체크아웃 후 라이브러리 미인식 문제가 반복되었다. 이를 방지하기 위해 4.12 분기에서는 프로젝트 구동과 빌드에 필요한 파일을 관리 대상으로 포함하였다.

---

### 6.1 SVN 관리 대상

4.12 분기에서 SVN 관리 대상으로 설정한 항목은 다음과 같다.

```text
- Logical
- Physical
- *.apj 프로젝트 파일
- batch.bat
- Logical / Physical 내부 구성 파일 전체
- 프로젝트 내부의 .pkg 파일
- 프로젝트 참조 및 라이브러리 관련 파일
```

`Logical` 폴더는 실제 프로그램의 논리 구성과 소스 구조를 포함하므로 관리 대상에 포함하였다.

`Physical` 폴더는 하드웨어 구성, I/O 구성, 물리 장치 구성과 관련된 정보를 포함하므로 관리 대상에 포함하였다.

`*.apj` 파일은 Automation Studio 프로젝트를 여는 기준 파일이므로 관리 대상에 포함하였다.

`batch.bat` 파일은 프로젝트 실행 또는 빌드 보조에 필요한 배치 파일로 판단되어 관리 대상에 포함하였다.

특히 기존 문제였던 라이브러리 미인식 문제를 방지하기 위해, 프로젝트 내부의 `.pkg` 파일과 라이브러리 참조에 필요한 파일들도 관리 대상으로 포함하였다. 이를 통해 신규 작업자가 SVN에서 프로젝트를 체크아웃했을 때, 별도의 수동 라이브러리 추가 없이 동일한 프로젝트 상태를 재현할 수 있도록 하였다.

---

### 6.2 SVN 관리 제외 대상

반대로 다음 항목은 SVN 관리 대상에서 제외하였다.

```text
- Binaries
- Diagnosis
- Temp
```

해당 폴더들은 프로젝트 원본이라기보다 빌드, 진단, 임시 작업 과정에서 생성되는 파생 파일 성격이 강하다.

`Binaries`는 빌드 결과물이며, 소스와 프로젝트 설정을 기준으로 재생성 가능한 파일이다.

`Diagnosis`는 진단 관련 산출물 성격이 강하므로 프로젝트 원본 관리 대상에서 제외하였다.

`Temp`는 임시 파일 저장 영역이므로 SVN에 포함하지 않는 것이 적절하다.

따라서 4.12 분기에서는 프로젝트 재현에 필요한 원본 파일과 설정 파일은 관리 대상으로 포함하고, 빌드 또는 실행 과정에서 생성되는 파생 파일은 제외하는 방식으로 관리 범위를 설정하였다.

---

## 7. 4.12 분기 관리 가이드라인

### 7.1 체크아웃 즉시 빌드 가능한 상태 유지

4.12 분기에서는 SVN에서 프로젝트를 새로 체크아웃했을 때, 별도의 수동 설정 없이 바로 열리고 빌드 가능한 상태를 목표로 한다.

이를 위해 다음 항목을 확인해야 한다.

```text
- 라이브러리 참조 누락 여부
- 프로젝트 경로 설정 정상 여부
- 빌드 설정 정상 여부
- 필수 설정 파일 포함 여부
- 불필요한 로컬 경로 의존성 여부
```

---

### 7.2 버전관리 포함/제외 기준 명확화

모든 파일을 SVN에 포함하는 것도 문제이고, 필요한 파일을 ignore 처리하는 것도 문제다.

따라서 파일별로 다음 기준을 적용한다.

```text
SVN 포함 대상:
- 소스 코드
- 프로젝트 설정 파일
- 빌드에 필요한 라이브러리 참조 파일
- Logical / Physical 구성 파일
- 장비 옵션 및 설정에 필요한 파일
- 체크아웃 후 동일 환경 재현에 필요한 파일

SVN 제외 대상:
- 자동 생성 TEMP 파일
- 불필요한 빌드 산출물
- 진단 산출물
- 작업자 로컬 환경에만 필요한 파일
- 개인 PC 경로에 종속된 파일
- 재생성 가능한 캐시성 파일
```

단, Diagnosis 관련 파일이나 바이너리 파일은 무조건 포함하지 않고, 실제 빌드 및 프로젝트 재현에 필요한지 여부를 기준으로 판단한다.

---

### 7.3 라이브러리 및 참조 구조 고정

라이브러리 누락 문제를 방지하기 위해 4.12 분기에서는 라이브러리와 참조 구조를 명확히 고정해야 한다.

확인 항목은 다음과 같다.

```text
- 프로젝트에 필요한 라이브러리 목록 정리
- 각 라이브러리의 참조 위치 확인
- SVN 체크아웃 후 라이브러리 인식 여부 확인
- 작업자 PC별 수동 추가가 필요한 항목 제거
- 상대경로 또는 표준 경로 사용 여부 확인
```

---

### 7.4 커밋 메시지 기준

향후 커밋 메시지는 수정 내용을 추적할 수 있도록 구분하여 작성한다.

권장 형식은 다음과 같다.

```text
[BUGFIX] 실제 오류 수정
[UPDATE] 기능 추가 또는 리소스 추가
[REFACTOR] 구조 개선 또는 코드 정리
[CONFIG] 프로젝트 설정 또는 라이브러리 참조 수정
[DOCS] 문서 또는 주석 정리
[HOTFIX] 현장 긴급 수정
```

예시는 다음과 같다.

```text
[BUGFIX] 냉각수 차단밸브 출력 조건 수정
[UPDATE] 태국어 리소스 추가
[REFACTOR] Userlog 정보 출력 구조 정리
[FIX] 서보온도 2존 표시 오류 수정
[CONFIG] 라이브러리 참조 구조 정리
```

---

### 7.5 수정 보고 작성 기준

중요 수정 건은 커밋 메시지만으로 끝내지 않고 간단한 수정 보고를 남기는 것이 좋다.

수정 보고에는 다음 항목을 포함한다.

```text
1. 문제 증상
2. 발생 조건
3. 원인 분석
4. 수정 위치
5. 수정 내용
6. 영향 범위
7. 확인 결과
```

특히 현장 장비와 관련된 출력, 안전 조건, 옵션 기능 수정은 반드시 판단 근거를 남기는 것이 좋다.

## 8. 4.12 분기 생성 시 점검 항목

4.12 신규 분기 생성 시 다음 항목을 점검한다.

```text
1. SVN 체크아웃 후 프로젝트 정상 오픈 여부
2. 라이브러리 자동 인식 여부
3. 추가 수동 설정 없이 빌드 가능 여부
4. TEMP 및 불필요 산출물 제외 여부
5. Diagnosis 관련 파일 제외 여부
6. Binaries 폴더 제외 여부
7. Logical / Physical 폴더 관리 대상 포함 여부
8. *.apj 파일 관리 대상 포함 여부
9. .pkg 및 라이브러리 참조 파일 포함 여부
10. 프로젝트 참조 경로의 로컬 PC 의존 여부
11. 기존 4.3 주요 수정 사항 반영 여부
12. 현장 버그 수정 이력 누락 여부
13. 신규 작업자 기준 재현 가능성 확인
```

---

## 9. 결론

기존 4.3 버전의 핵심 문제는 C 코드 수정 이력은 SVN에 존재했지만, 실제 빌드와 실행에 필요한 라이브러리, 참조 파일, 프로젝트 환경이 일관되게 관리되지 않았다는 점이다.

이로 인해 같은 SVN 버전을 체크아웃해도 작업자마다 다른 프로젝트 상태가 되었고, 체크아웃 후 라이브러리를 다시 추가하거나 환경을 수동으로 맞춰야 하는 문제가 반복되었다.

4.12 신규 분기는 이러한 문제를 정리하고, 향후 유지보수 가능한 기준점을 새로 만드는 작업이다.

따라서 4.12에서는 다음 기준을 반드시 지켜야 한다.

```text
- 체크아웃 즉시 빌드 가능
- 라이브러리 및 참조 구조 누락 방지
- Logical / Physical / apj / pkg 파일 관리 대상 포함
- Binaries / Diagnosis / Temp 파생 파일 제외
- 작업자 로컬 환경 의존 제거
- 기능 수정 시 판단 근거 기록
- 옵션 기능은 최종 출력 구조까지 확인
- 커밋 메시지 및 수정 보고 기준 정립
```

이 기준이 지켜져야 4.12 분기는 단순 버전 업그레이드가 아니라, 기존 4.3의 관리 실패를 보완한 안정적인 기준 분기로 기능할 수 있다.
