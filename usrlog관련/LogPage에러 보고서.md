# 로그 단위 불일치 및 값 왜곡 문제 분석 보고서

## 1. 개요

본 시스템에서 기록되는 로그 값이 실제 장비 설정값 또는 기대값과 일치하지 않는 문제가 발생하였다.
코드 분석 결과, 로그 값과 단위는 **일관된 단일 변환 체계로 처리되지 않고**, `displayunit-stext` 기반 단위 해석과 `UserId` 기반 예외 보정이 동일 처리 흐름 안에 함께 존재한다. 또한 다수의 `UserId` 조건 분기를 통해 값과 단위가 개별적으로 재설정되고 있다.

---

## 2. 주요 증상

### 2.1 로그 값과 실제 설정값 불일치

* HMI 또는 장비에서 설정된 값과 로그에 기록된 값이 다르게 표시됨
* 특정 항목은 값이 과도하게 크거나 작게 기록됨

### 2.2 단위(% / bar / 기타)의 비일관성

* 동일 계열 데이터임에도 이벤트에 따라 `%`, `bar` 등의 단위 적용이 다르게 나타남
* 단위 문자열과 실제 값 스케일이 불일치

---

## 3. 원인 분석

### 3.1 확인된 사실: 단위 해석과 UserId 기반 재보정이 동시에 존재함

코드는 이벤트에서 `displayunit-stext`를 읽어 `charUnitText`로 변환한 뒤, 문자열 비교를 통해 `ConvMulti`, `ConvOffset`를 결정한다. 같은 처리 흐름 안에서 이후 다시 `UserId` 조건문에 따라 `valueLog.NewValue`, `valueLog.OldValue`를 재계산하고, 단위 문자열도 `%` 또는 `bar`로 다시 덮어쓴다.

즉, 현재 구조에서는

* 이벤트 단위 문자열 기반 처리
* `UserId` 기반 예외 처리

가 동일 함수 안에서 함께 수행된다. 이로 인해 최종 로그 값과 단위는 단일 규칙이 아니라, 여러 분기 조건의 적용 결과로 결정된다.

---

### 3.2 확인된 사실: 값 스케일링 기준이 여러 구조체에 분산되어 있음

코드에서는 `%` 환산 시 다음과 같이 서로 다른 기준값을 사용한다.

* `CU_FIX.NEG.SvMax1Vis`
* `CU_FIX.POS.SvMax1Vis`
* `EJ_FIX.NEG.SvMaxVis`
* `EJ_FIX.POS.SvMaxVis`
* `EA_FIX[S_MH].NEG.SvMaxVis`
* `IU_FIX.NEG.SvMaxVis`
* `IU_FIX.POS.SvMaxVis`

즉, 로그 값 환산 기준이 단일 테이블이나 공통 함수가 아니라, `UserId` 조건에 따라 각기 다른 구조체 필드에 직접 연결되어 있다.

결과적으로, 동일 로그 처리 함수 내부에서 값 변환 기준이 분산되어 있으며, 변환 정책이 한 곳에서 통합 관리되지 않는다.

---

### 3.3 확인된 사실: 단위 문자열은 UTF-16 값을 1바이트 문자 배열로 변환해 비교함

코드는 `VCEV_Utf16Get(..., "displayunit-stext", ...)`로 단위 문자열을 읽은 뒤, 이를 `charUnitText`에 1바이트 문자로 복사한다. 이 과정에서 `~`(126)보다 큰 문자는 강제로 `~`로 치환하도록 되어 있다. 이후 `" in"`, `" in/s"`, `" ~F"`, `" Kg/cm~"`, `" psi"` 문자열과 비교해 변환 계수를 선택한다. 

따라서 보고서에는 다음처럼 쓰는 것이 맞다.

> 단위 문자열 처리는 UTF-16 원문을 그대로 비교하는 방식이 아니라, 1바이트 문자 배열로 변환한 뒤 일부 문자를 치환하여 비교하는 방식으로 구현되어 있다. 

이건 추정이 아니라 코드에서 직접 확인되는 사실이다.

---

### 3.4 확인된 사실: 조건 분기가 누적되어 있고 중복 조건이 존재함

`bar` 변환 조건식에는 `(UserId==342 || UserId==345)`가 동일 조건문 내에 두 번 포함되어 있다. 또한 여러 조건식이 긴 OR 형태로 연결되어 있으며, 값 변환과 단위 문자열 변경이 각각의 분기 안에서 개별적으로 수행된다. 

따라서 다음은 명확히 보고 가능하다.

* 조건 분기는 장문 OR 형태로 누적되어 있다
* 동일 조건의 중복 삽입이 존재한다
* 조건식 내부에서 값과 단위가 직접 수정된다 

---

### 3.5 확인된 사실: 로그 처리 흐름 안에 설비 상태 갱신 로직이 함께 들어가 있음

동일 처리 흐름 안에서 로그 값 보정뿐 아니라 다음 동작도 함께 수행된다.

* `MMI_DATA.DatCpy.HeaterDataChanged = 1`
* `REC_DATA.HC.EXTOnoff`, `REC_DATA.HC.EXTOnoff_1` 갱신
* CMS 통신 모드에 따라 `Modbus_Bit[]`, `Tcp_Bit[]` 적용

즉, 로그 이벤트 처리와 설비 상태 반영이 동일 블록 안에 결합되어 있다. 

이 역시 추정이 아니라 코드상 직접 확인되는 구조다.

---

## 4. 잠재적 문제점

### 4.1 잠재적 문제점: 비트 상태를 `+=`, `-=`로 처리함

코드는 외부 온도 선택 상태 갱신 시 `EXTOnoff`, `EXTOnoff_1`에 대해 비트 연산이 아니라 `+=`, `-=`를 사용하고 있다. 

이 구현 방식은 상태값을 비트 마스크로 관리하는 의도와 다를 경우 오류를 유발할 수 있으므로, 구조적으로 취약한 방식으로 판단된다.
보고서에서는 다음처럼 적는 것이 적절하다.

> 비트성 상태값을 `+=`, `-=` 방식으로 갱신하고 있어, 상태 재적용이나 중복 처리 상황에 취약할 수 있는 구현 형태를 가지고 있다. 

---

### 4.2 잠재적 문제점: 스케일 계산 시 0 나눗셈 방어가 보이지 않음

`SvMax1Vis`, `SvMaxVis`를 분모로 사용하는 계산이 다수 존재하지만, 해당 코드 구간에서는 분모가 0인지 확인하는 방어 조건이 보이지 않는다.

따라서 확정적으로 “0 나눗셈이 발생했다”고 쓸 수는 없지만, 아래처럼는 쓸 수 있다.

> 값 환산 시 분모로 사용되는 기준값에 대한 방어 조건이 해당 구간에서 확인되지 않으므로, 기준값 검증이 별도로 보장되지 않는 구조이다.

---

### 4.3 잠재적 문제점: `pow()` 기반 실수 계산을 사용함

`dPoint.uiDevide`, `dPoint.uiDevide2` 계산에 `pow(10.0f, ...)`가 사용된다. 

이 부분은 “오류가 발생했다”가 아니라 다음처럼 표현하는 것이 적절하다.

> 자릿수 계산에 실수 기반 `pow()`를 사용하고 있어, 정수 중심 로그 처리 체계와 계산 방식이 분리되어 있다. 

---

## 5. 종합 정리

본 문제는 특정 계산식 하나의 오류라기보다, 다음과 같은 코드 구조에서 직접 확인되는 사항들이 결합된 결과이다.

1. 단위 문자열 해석과 `UserId` 기반 재보정이 동시에 존재함
2. 값 스케일 기준이 여러 구조체 필드에 분산되어 있음
3. 조건 분기가 장문 OR 형태로 누적되어 있음
4. 동일 조건의 중복 삽입이 존재함
5. 로그 처리와 설비 상태 갱신 로직이 한 흐름에 결합되어 있음

즉,

> **단위 정의, 값 환산, 상태 반영이 분리된 계층으로 관리되지 않고, 동일 처리 흐름 안에서 개별 조건문 중심으로 결합된 구조가 현재 문제의 직접적인 배경이다.**

---

## 6. 개선 방향

### 6.1 단위/스케일 처리의 단일화

현재처럼

```c
if(UserId == ...)
```

기반으로 값과 단위를 개별 수정하는 방식 대신,

```c
UserId → {단위, 스케일, 처리 방식}
```

형태의 통합 매핑 구조로 전환할 필요가 있다.

### 6.2 값 변환과 단위 설정의 통합

현재는 단위 문자열 해석 후 다시 `UserId` 예외 처리에서 값을 재보정하고 단위를 덮어쓰고 있으므로, 최종 저장 직전의 변환 경로를 단일화해야 한다.

### 6.3 상태 처리 분리

로그 처리와 상태 갱신(`HeaterDataChanged`, `EXTOnoff`)은 별도 계층으로 분리해야 한다.

### 6.4 조건식 정리

중복 조건 제거, 범위 조건 괄호 명시, 상수화가 필요하다.

---

## 7. 결론

현재 로그 값 오류 문제는 코드에서 직접 확인되는 다음 구조와 연결되어 있다.

* 단위 문자열 기반 처리와 `UserId` 기반 예외 처리의 병존
* 다수 기준값에 의존한 분산형 스케일링
* 중복된 조건 분기
* 로그 처리와 상태 갱신의 결합

따라서 본 문제는 단순 계산 오류로 보기보다,

> **로그 단위와 값 변환 정책이 단일 체계로 통합되지 않은 구현 구조에서 발생한 문제**

로 정리하는 것이 적절하다.
