# 로그 단위 불일치 및 값 왜곡 문제 개선 보고서

## 1. 개요

본 개선 작업은 로그 데이터에서 발생하던 단위 불일치 및 값 왜곡 문제를 해결하기 위해 수행되었다.
기존 코드에서는 로그 값 계산 과정에 복수의 기준이 혼재되어 있었으며, 이로 인해 동일 항목이라도 처리 경로에 따라 값과 단위가 달라지는 문제가 발생하고 있었다.

본 작업은 기존 코드 베이스를 유지한 상태에서, **로그 값 및 단위 계산 구조를 정리하여 일관된 계산 방식으로 통합**하는 것을 목표로 수행되었다.

---

## 2. 기존 문제점

### 2.1 값 계산 기준의 중복 및 재개입

기존 구조에서는 로그 값이 다음과 같은 흐름으로 처리되고 있었다.

* 단위 문자열(`displayunit-stext`) 기반 1차 변환
* 이후 `UserId` 조건에 따라 값 재계산
* 단위 문자열 재설정

즉, 하나의 값에 대해 **두 개 이상의 기준이 순차적으로 개입하는 구조**였으며,
이로 인해 최종 값이 어떤 기준으로 계산되었는지 일관성이 보장되지 않았다.

---

### 2.2 변환 로직의 분산

`%`, `bar` 등의 변환 규칙이 `_CYCLIC` 내부 여러 위치에 분산되어 있었으며,
다음과 같은 형태로 구성되어 있었다.

* `if (UserId == ...)`
* `if (UserId >= ... && UserId <= ...)`

이로 인해:

* 동일 규칙이 여러 위치에 중복 정의됨
* 일부 조건 누락 가능성 존재
* 유지보수 시 영향 범위 파악 어려움

---

### 2.3 값 계산과 부수 처리의 혼재

로그 값 계산과 동시에 다음과 같은 기능들이 동일 블록에서 수행되고 있었다.

* HeaterDataChanged 갱신
* 외부온도 설정 반영
* CMS 통신 상태 처리
* HMI IP 반영

이로 인해:

* 로그 계산 로직의 가독성 저하
* 값 계산 흐름 추적 어려움

---

## 3. 개선 내용

## 3.1 로그 값 계산 구조 단일화

기존에는 값이 여러 단계에서 재계산되던 구조를 다음과 같이 정리하였다.

* 로그 값 계산을 `usrlog_apply_conversion()` 함수로 통합
* 해당 함수에서 **최종 값과 단위를 한 번만 결정**

즉,

> 하나의 로그 값은 하나의 경로에서만 계산되도록 구조를 변경하였다.

---

## 3.2 변환 규칙의 구조화

기존에 분산되어 있던 `UserId` 기반 조건문을 정리하여:

* `usrlog_get_rule()` 함수로 변환 규칙을 통합
* `%`, `bar`, 기본 단위 변환을 명확히 구분

변환 흐름은 다음과 같이 정리되었다.

```
UserId → 변환 규칙 선택 → 값 계산 → 단위 결정
```

이를 통해:

* 변환 기준의 단일화
* 조건 중복 제거
* 규칙 변경 시 수정 위치 최소화

---

## 3.3 기본 단위 변환 로직 유지 및 정리

기존 코드에서 사용되던 단위 문자열 기반 변환은 유지하되,
다음과 같이 정리하였다.

* UTF-16 단위 문자열 → char 변환
* 단위 문자열에 따른 변환 계수 적용
* decimal point 기준 스케일 적용

기본 단위 변환은 `%`, `bar` 예외 규칙과 분리하여 **독립된 처리 흐름으로 유지**하였다.

---

## 3.4 `%` 변환 스케일 보정

속도 관련 `%` 로그 값에서 자릿수 오류가 발생하고 있었다.

예:

* 입력값 130 → 로그 130 (기존)
* 기대값 13
* 입력값 135 → 로그 135 (기존)
* 기대값 13.5

원인:

* `%` 변환 시 `*1000 / svMax` 스케일 사용

개선:

* `*100 / svMax`로 수정

이를 통해 로그 값이 기대하는 자릿수 형태로 정상 출력되도록 보정하였다.

---

## 3.5 불필요한 재계산 로직 제거

기존 `_CYCLIC` 내부에 존재하던 다음 로직을 제거하였다.

* `%`, `bar` 재계산 블록
* 단위 문자열 재설정(`strcpy`) 반복 처리
* 동일 값에 대한 중복 변환

이로 인해:

* 계산 과정 단순화
* 값 왜곡 발생 가능성 제거

---

## 3.6 rounding 처리 정리

기존 코드에서 사용되던 `round1()` 함수는 현재 프로젝트에서 링크되지 않아 오류가 발생하였다.

이에 따라:

* `round1()` → `round0()`으로 변경
* 기존 프로젝트에서 사용 중인 라이브러리 기준에 맞게 정리

이를 통해 빌드 안정성을 확보하였다.

---

## 3.7 부수 처리 로직 유지

다음 로직은 장비 동작과 직접 연결되므로 유지하였다.

* HeaterDataChanged 처리
* 외부온도 및 존 이름 반영
* CMS 통신 상태 처리
* HMI IP 설정

단, 로그 값 계산 이후에 수행되도록 하여 **계산 로직과 기능 로직을 분리**하였다.

---

## 4. 개선 결과

개선 전:

```
단위 해석 → 값 계산 → UserId 재계산 → 단위 재설정 → 로그 저장
```

개선 후:

```
UserId → 변환 규칙 선택 → 단일 계산 → 로그 저장
                     ↓
                 부수 처리
```

결과적으로:

* 값 계산 기준 단일화
* 단위 일관성 확보
* 중복 로직 제거
* 유지보수성 향상

---

## 5. 결론

본 개선을 통해 로그 값 계산 구조에서 발생하던 **중복 변환 및 기준 재개입 문제를 제거**하고,
로그 값과 단위가 **단일한 규칙에 의해 결정되는 구조로 정리**하였다.

또한 `%` 스케일 오류 및 rounding 문제를 함께 수정하여, 실제 기대값과 로그 값 간의 불일치를 해소하였다.

본 구조는 기존 코드 베이스를 유지하면서도,
추가적인 항목 확장 및 유지보수가 가능한 형태로 정리된 상태이다.
