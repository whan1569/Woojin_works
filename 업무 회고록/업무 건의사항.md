현재 문제는 단순히 일이 많다는 것보다, 코드 업무 프로세스와 테스트 기준이 명확히 정립되어 있지 않다는 점이 크다고 생각합니다.

현재 DL, TH, 현장 대응, 외근, FlexConfig 같은 업무가 여러 방향에서 들어오고 있습니다.

각 업무는 단순 수정이 아니라 기종별 히스토리, 기존 코드 흐름, 변경 목적, 관련 변수, 장비 동작 조건, 테스트 기준을 다시 확인해야 하는 업무입니다.

하드웨어나 사출기 관점에서 가설을 제시할 수 있는 분들은 있지만, 그 가설을 실제 코드에서 검증하고, 수정 가능 여부와 영향 범위, 테스트 기준까지 판단할 수 있는 인원은 제한적입니다.

그러다 보니 여러 가설이나 의견이 들어와도 결국 코드상 검증, 수정 범위 판단, 테스트 기준 설정은 제 쪽으로 많이 집중됩니다.

또 방법론이나 방향성을 제안하는 사람은 많지만, 그 방법이 실제 코드의 어떤 문제를 해결하는지, 어디를 어떻게 바꿔야 하는지, 무엇을 테스트하면 완료인지까지 제시되는 경우는 많지 않습니다.

즉, 방법론 제안과 실제 해결책 도출이 구분되지 않는 문제가 있습니다.

그리고 코드 업무는 바로 타이핑하는 시간만 일이 아닙니다. 기존 코드를 읽고, 문제를 쪼개고, 동작을 시뮬레이션하고, 필요한 내용을 다시 공부하는 시간이 필요합니다.

그런데 이 과정은 외부에서 보기에는 단순히 화면을 보거나 생각하는 시간처럼 보일 수 있어서, 실제 업무 중인데도 일이 없는 시간처럼 판단되는 경우가 있습니다.

또 하나 중요한 문제는 테스트와 현장 적용 기준입니다.

사출기는 금형, 옵션, 고객 현장 조건이 모두 다르기 때문에 현장에 나가서 어느 정도 디버깅하거나 커스터마이징하는 과정은 불가피하다고 생각합니다.

다만 문제는 내부에서 기준이 되는 표준점이 명확하지 않다는 점입니다.

공통 기능 기준으로 어디까지는 내부에서 기본 검증되어야 하는지, 어디부터가 기종별 차이인지, 어디부터가 고객 현장 조건에 따른 커스터마이징인지, 어디부터가 추가 요구사항인지가 구분되어야 합니다.

현재는 이 경계가 불명확한 경우가 있어, 현장에서 문제가 발생했을 때 그것이 기본 기능 검증 부족인지, 기종별 차이인지, 금형이나 고객 조건에 따른 커스터마이징 영역인지 판단하는 데 시간이 많이 들어갑니다.

이렇게 기준점이 없으면 문제가 발생할 때마다 다시 수정하고, 버전을 여러 개 오가며 현장에서 수렴시키는 방식이 반복됩니다.

겉으로는 계속 대응하고 수정하는 것처럼 보일 수 있지만, 실제로는 표준 검증 기준이 부족해서 같은 문제를 반복해서 확인하고 재작업하는 구조가 될 수 있습니다.

특히 재작업과 실질적 진척이 구분되지 않으면, 여러 버전을 오가며 계속 수정한 일이 처음부터 충분히 검토하고 안정적으로 처리한 일보다 더 많이 일한 것처럼 보이는 왜곡이 생길 수 있습니다.

이 상태가 계속되면 충분한 분석 없이 업무가 전환되고, 그 결과 재작업, 테스트 누락, 원인 추적 어려움이 생길 수 있습니다.

그래서 앞으로는 코드 업무를 증상 확인, 가설 정리, 코드 원인 분석, 수정안 작성, 영향 범위 확인, 내부 표준 검증, 현장 커스터마이징, 최종 검증 완료 단계로 구분해서 진행할 필요가 있다고 봅니다.

그리고 코드 검증과 유지보수를 보조할 수 있는 신입 인력이라도 필요합니다.

처음부터 핵심 로직을 맡기는 것이 아니라, 코드 위치 정리, 변수 매핑, 변경 이력 정리, 테스트 체크리스트 작성, 재현 조건 기록 같은 업무만 맡아도 현재 병목을 줄이는 데 도움이 될 것 같습니다.

장기적으로는 회사 안에 기종별 코드 히스토리, 표준 검증 기준, 현장 커스터마이징 사례가 축적되기 때문에 특정 개인에게만 코드 검증과 현장 대응 경험이 몰리는 리스크도 줄일 수 있다고 생각합니다.
