1년 정도 업무를 진행하면서 느낀 구조적인 부분을 정리해서 말씀드리고 싶습니다.

먼저 현재 구조가 무조건 나쁘다고 생각하지는 않습니다. 실제로 업무를 하면서 여러 분들에게 도움을 받기도 했고, 저 역시 존중을 받고 있다고 느끼는 부분도 많습니다.

사출기 업무 특성상 하드웨어, 사출 공정, 현장 조건, 코드가 서로 연결되어 있기 때문에 기계적 관점과 코드 관점이 완전히 분리되면 오히려 소통의 벽이 생길 수 있습니다. 현장이나 하드웨어 관점에서 증상과 가능성을 제시하고, 코드 담당자가 그 가설을 코드 흐름과 변수 상태, 출력 조건 기준으로 검증하는 현재 방식 자체는 장점이 있다고 생각합니다.

다만 업무를 진행하면서 반복적으로 느끼는 문제는, 코드 검증을 수행할 수 있는 인원이 제한적이라는 점입니다.

현재 제가 맡거나 지원하는 업무는 DL, TH, VH 보조, 현장 대응, 외근, FlexConfig 등 여러 영역에 걸쳐 있습니다. 각 업무는 단순히 코드 몇 줄을 수정하는 것이 아니라, 기종별 히스토리, 기존 코드 흐름, 변경 목적, 관련 변수, 장비 동작 조건, 테스트 기준을 다시 확인해야 하는 형태입니다.

문제를 정확히 파악하려면 기계적 가설뿐만 아니라, 그 가설이 코드상으로 가능한지, 어떤 조건에서 발생할 수 있는지, 어떤 변수나 출력과 연결되는지 확인할 수 있어야 합니다.

그런데 코드 검증이 가능한 인원이 부족하다 보니, 문제 파악 단계부터 정확한 원인 후보를 좁히기 어렵고, 여러 가설이나 의견이 들어와도 결국 코드상 검증, 수정 가능 여부 판단, 영향 범위 확인, 테스트 기준 설정이 특정 인원에게 집중되는 구조가 됩니다.

또 방법론이나 방향성을 제안하는 사람은 많지만, 그 방법이 실제 코드의 어떤 문제를 해결하는지, 어디를 어떻게 바꿔야 하는지, 무엇을 테스트하면 완료인지까지 정리되는 경우는 많지 않습니다.

즉, 방법론 제안과 실제 해결책 도출이 구분되지 않는 경우가 있습니다.

방법론이나 방향성 제안도 필요하지만, 실제 해결책이 되려면 현재 코드의 어느 부분이 원인인지, 어떤 파일과 변수에 반영해야 하는지, 변경 영향 범위가 어디까지인지, 어떤 테스트를 통과하면 완료인지까지 정리되어야 합니다.

현재는 이 과정이 명확히 나뉘어 있지 않다 보니, 여러 의견이 들어와도 다시 코드 구조를 확인하고, 실제로 가능한지 검증하고, 수정 범위와 테스트 기준을 잡는 시간이 반복적으로 발생합니다.

그리고 코드 업무는 바로 타이핑하는 시간만 일이 아닙니다.

기존 코드를 읽고, 문제를 쪼개고, 동작 흐름을 시뮬레이션하고, 필요한 내용을 다시 확인하거나 공부하는 시간이 필요합니다. 이 과정은 외부에서 보기에는 단순히 화면을 보거나 생각하는 시간처럼 보일 수 있지만, 실제로는 수정 방향을 정하고 오류를 줄이기 위한 필수 과정입니다.

특히 여러 TASK가 동시에 들어오면 각 업무의 배경, 이전 변경 목적, 관련 코드 흐름, 테스트 기준을 다시 복기해야 합니다. 이 복기 시간이 반복적으로 길어지는 것이 현재 가장 큰 병목 중 하나입니다.

이 부분이 외부에서 잘 보이지 않다 보니, 서로 대화할 때 오해가 생기는 경우도 있습니다.

저는 실제로 문제를 정리하거나 시뮬레이션하고 있는 상태인데, 겉으로는 바로 결과물이 나오지 않기 때문에 일이 멈춰 있는 것처럼 보일 수 있습니다. 반대로 업무를 요청하는 입장에서도 코드 검증에 어느 정도 시간이 필요한지 정확히 보기 어렵기 때문에, 답답하게 느껴질 수 있다고 생각합니다.

그래서 저도 미안함을 느낄 때가 있고, 반대로 충분히 검토해야 하는 업무가 단순 대기처럼 보일 때는 답답함을 느낄 때도 있습니다. 이것은 누가 잘못했다기보다, 코드 업무의 분석 과정과 검증 상태가 외부에서 잘 보이지 않기 때문에 생기는 문제라고 생각합니다.

또 하나 중요한 부분은 테스트와 현장 적용 기준입니다.

사출기는 금형, 옵션, 고객 현장 조건이 모두 다르기 때문에 현장에 나가서 어느 정도 디버깅하거나 커스터마이징하는 과정은 불가피하다고 생각합니다.

다만 내부에서 기준이 되는 표준점이 명확하지 않으면, 어디까지가 내부에서 기본 검증되어야 하는 부분인지, 어디부터가 기종별 차이인지, 어디부터가 고객 현장 조건에 따른 커스터마이징인지, 어디부터가 추가 요구사항인지 구분하기 어렵습니다.

이 경계가 불명확하면 현장에서 문제가 발생했을 때 그것이 기본 기능 검증 부족인지, 기종별 차이인지, 금형이나 고객 조건에 따른 커스터마이징 영역인지 판단하는 데 시간이 많이 들어갑니다.

그 결과 문제가 발생할 때마다 다시 수정하고, 버전을 여러 개 오가며 현장에서 수렴시키는 방식이 반복될 수 있습니다.

겉으로는 계속 대응하고 수정하는 것처럼 보일 수 있지만, 실제로는 표준 검증 기준이 부족해서 같은 문제를 반복해서 확인하고 재작업하는 구조가 될 수 있습니다.

특히 재작업과 실질적 진척이 구분되지 않으면, 여러 버전을 오가며 계속 수정한 일이 처음부터 충분히 검토하고 안정적으로 처리한 일보다 더 많이 일한 것처럼 보이는 왜곡이 생길 수 있습니다.

이 상태가 계속되면 충분한 분석 없이 업무가 전환되고, 그 결과 재작업, 테스트 누락, 원인 추적 어려움이 생길 수 있습니다.

그래서 앞으로는 코드 업무를 증상 확인, 가설 정리, 코드 원인 분석, 수정안 작성, 영향 범위 확인, 내부 표준 검증, 현장 커스터마이징, 최종 검증 완료 단계로 구분해서 진행할 필요가 있다고 봅니다.

그리고 필요한 것은 기계 쪽과 코드 쪽을 완전히 분리하는 것이 아니라, 현재 협업 구조를 유지하되 코드 검증을 함께 수행할 수 있는 인력을 보강하는 것이라고 생각합니다.

단순히 인원을 늘려달라는 의미보다는, 코드 흐름을 같이 읽고, 가설을 코드상에서 검증하고, 수정 영향 범위와 테스트 기준을 함께 판단할 수 있는 사람이 필요하다는 의미입니다.

기계적 가설을 제시하는 역할과 코드상 가능성을 검증하는 역할이 서로 소통하면서 움직이되, 코드 검증 부담이 특정 인원에게만 몰리지 않도록 구조를 보완할 필요가 있습니다.

장기적으로는 회사 안에 기종별 코드 히스토리, 표준 검증 기준, 현장 커스터마이징 사례가 축적되어야 한다고 생각합니다.

그래야 특정 개인에게만 코드 검증과 현장 대응 경험이 몰리는 리스크를 줄이고, 이후 비슷한 문제가 발생했을 때도 더 빠르고 안정적으로 대응할 수 있을 것입니다.
