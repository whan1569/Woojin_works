## 냉각수 차단밸브 출력 확인 및 수정

### 1. 작업 배경

현장 확인 중 냉각수 차단밸브 출력이 필요한 조건에서 정상적으로 열리지 않는 현상이 확인되었다.
초기에는 해당 출력이 단순히 `DO_18_1` 디지털 출력으로 보였기 때문에, 조건 만족 시 `DO_18_1`을 직접 ON 시키는 방식으로 접근하였다.

초기 검토 방향은 다음과 같았다.

```c
if (MA_STAT.AbWaterFlow_Use)
{
    DO_18_1 = 1;
}
```

즉, `MA_STAT.AbWaterFlow_Use` 조건이 활성화되면 냉각수 차단밸브에 해당하는 `DO_18_1` 출력을 직접 살리는 방식이었다.

---

### 2. 1차 시도의 한계

그러나 `DO_18_1`을 사이클 내부에서 직접 ON 시키는 방식은 정상적으로 유지되지 않았다.
원인은 `DO_18_1`이 단순 독립 출력처럼 사용되는 것이 아니라, 이후 출력 처리 구조 또는 래더 매핑에서 다른 내부 변수에 의해 최종 결정되는 구조였기 때문이다.

즉, 중간에서 `DO_18_1 = 1`을 넣더라도, 이후 출력 처리 과정에서 `DO_18_1` 값이 다시 갱신되거나 덮어써질 수 있었다.

따라서 문제의 핵심은 `DO_18_1` 자체를 강제로 ON 시키는 것이 아니라, `DO_18_1`을 최종적으로 구동하는 내부 조건 변수를 찾아 수정하는 것이었다.

---

### 3. 출력 담당 변수 확인

래더 및 출력 매핑을 확인한 결과, `DO_18_1`은 직접 조건을 받아 동작하는 출력이 아니라 `DO_WaterFlow_Fixplate` 변수에 의해 최종 구동되는 구조로 확인되었다.

구조는 다음과 같이 정리할 수 있다.

```text
DO_WaterFlow_Fixplate
        ↓
래더 접점
        ↓
DO_18_1
        ↓
고정측 냉각수 차단밸브 출력
```

코드상에서도 냉각수 차단밸브 관련 변수는 다음과 같이 별도 정의되어 있었다.

```c
DO_WaterFlow_Fixplate = ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP];
DO_WaterFlow_Moveplate = (MMI_DATA.Mode == MODE_SEMI_AUTO || MMI_DATA.Mode == MODE_AUTO) ? 1 : 0;
```

따라서 `DO_18_1`을 직접 제어하는 방식은 구조적으로 적절하지 않았고, 실제 수정 대상은 `DO_WaterFlow_Fixplate` 조건임을 확인하였다.

---

### 4. 수정 방향 판단

최종 판단은 다음과 같다.

```text
수정 대상: DO_18_1 직접 제어 아님
실제 수정 대상: DO_WaterFlow_Fixplate 조건
```

기존에는 고정측 냉각수 차단밸브가 금형온도제어 그룹의 히터 ON 상태를 기준으로 동작하고 있었다.
하지만 현장에서 필요한 조건은 `MA_STAT.AbWaterFlow_Use`가 활성화되었을 때도 냉각수 차단밸브가 열려야 하는 것이었다.

따라서 단순히 `DO_18_1`을 강제로 ON 시키는 것이 아니라, `DO_WaterFlow_Fixplate` 조건에 `MA_STAT.AbWaterFlow_Use`를 반영하는 방향으로 수정하는 것이 맞다고 판단하였다.

예상 수정 방향은 다음과 같다.

```c
DO_WaterFlow_Fixplate =
    ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP]
    || MA_STAT.AbWaterFlow_Use;
```

이렇게 수정하면 `MA_STAT.AbWaterFlow_Use` 조건이 활성화되었을 때 `DO_WaterFlow_Fixplate`가 ON 되고, 래더를 통해 최종적으로 `DO_18_1` 출력이 정상적으로 ON 된다.

---

### 5. 정리

이번 검토의 핵심은 출력 주소를 직접 제어하는 것이 아니라, 해당 출력을 담당하는 내부 제어 변수를 찾아 수정해야 한다는 점이었다.

초기에는 `DO_18_1`을 직접 ON 시키는 단순한 방식으로 접근했으나, 실제 구조상 `DO_18_1`은 `DO_WaterFlow_Fixplate`를 통해 최종 출력되는 구조였다.
따라서 직접 출력 제어 방식은 이후 출력 처리 과정에서 무효화될 수 있었고, 최종적으로는 `DO_WaterFlow_Fixplate` 조건에 `MA_STAT.AbWaterFlow_Use`를 반영하는 방향으로 수정 판단하였다.

결론적으로, 이번 작업은 단순 DO 출력 강제 방식에서 벗어나 실제 출력 담당 변수와 래더 매핑 구조를 확인한 뒤, 올바른 제어 지점에 조건을 추가한 사례로 정리할 수 있다.
