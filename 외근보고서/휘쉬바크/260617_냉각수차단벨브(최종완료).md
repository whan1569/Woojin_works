# tk2o.c 코어풀 및 냉각수 출력 로직 수정 보고서

## 1. 개요

본 보고서는 `tk2o.c` 출력 처리 로직 중 **코어풀 옵션 동작 타이밍**과 **냉각수 차단밸브 출력 조건**을 수정한 내용을 정리한 것이다.

기존 코어풀 자동/반자동 로직은 형폐완료 이후 `MA_STAT.AbCorePull_Flag`가 활성화되면 코어풀 동작이 진행될 수 있는 구조였다. 그러나 실제 요구사항은 코어풀이 형폐완료 직후나 노즐터치 완료 직후에 단독으로 동작하는 것이 아니라, **노즐터치 완료 후 사출 시작과 동시에 코어풀이 함께 동작하는 구조**이다.

이에 따라 본 수정에서는 `co_con.c`, `aut_main.c`, `tk1o.c`는 수정하지 않고, `tk2o.c` 내부에서 코어풀 요청 플래그의 실제 실행 타이밍을 제어하도록 변경하였다.

## 2. 수정 목적

### 2.1 코어풀 자동 동작 타이밍 변경

기존 자동/반자동 코어풀은 형폐완료 이후 `MA_STAT.AbCorePull_Flag`가 살아 있으면 코어풀 출력이 진행될 수 있는 구조였다.

변경 후에는 다음 순서로 동작하도록 수정하였다.

```text
형폐완료
→ 코어풀 요청 감지
→ 코어풀 요청 보류
→ 노즐터치 완료 확인
→ 사출 시작 확인
→ 코어풀 실행
```

즉, 코어풀은 형폐완료 직후 실행되지 않고, 노즐터치 완료 후에도 바로 실행되지 않으며, **사출 출력이 시작되는 시점에 맞춰 실행**되도록 변경하였다.

### 2.2 co_con.c 수정 없이 기존 코어축 상태머신 유지

`co_con.c`는 `MA_STAT.AbCorePull_Flag`를 기준으로 `S_CO1` 코어축을 시작하는 구조이다.

본 수정에서는 `co_con.c`의 상태머신을 수정하지 않고, `tk2o.c`에서 `MA_STAT.AbCorePull_Flag`의 발생 타이밍만 제어하였다.

이를 통해 기존 코어축 제어 구조는 유지하면서, 자동/반자동 운전 중 코어풀 시작 시점만 요구사항에 맞게 조정하였다.

### 2.3 기존 DO_5_3 출력 구조 유지

초기 검토 과정에서는 `DO_5_3` 출력 소유권을 완전히 정리하는 방안도 고려하였다.

그러나 최종 수정에서는 검증 범위 확대를 방지하기 위해 기존 구조를 유지하였다.

```text
CoreOpen()  → DO_5_3 = 1
CoreClose() → DO_5_3 = 0
```

즉, 이번 수정의 핵심은 출력 매핑 구조 변경이 아니라, **코어풀 출력이 켜지는 시점을 제어하는 것**이다.

### 2.4 냉각수 고정측 출력 조건 추가

기존 고정측 냉각수 출력은 금형히터 ON 상태만 기준으로 출력되었다.

```c
DO_WaterFlow_Fixplate = ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP];
```

변경 후에는 별도 냉각수 사용 요청 상태인 `MA_STAT.AbWaterFlow_Use`도 출력 조건에 포함하였다.

```c
DO_WaterFlow_Fixplate =
    ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP]
    || MA_STAT.AbWaterFlow_Use;
```

## 3. 기존 코드의 주요 문제점

### 3.1 코어풀 실행 기준이 형폐완료에 가까움

기존 자동/반자동 코어풀 로직은 형폐완료 상태에서 `MA_STAT.AbCorePull_Flag`가 활성화되면 코어풀 출력이 진행될 수 있었다.

문제는 `MA_STAT.AbCorePull_Flag`가 사출 시작 시점이 아니라, 자동 시퀀스상 형폐완료 이후에 먼저 세워지는 구조라는 점이다.

따라서 기존 구조에서는 다음과 같은 흐름이 발생할 수 있다.

```text
형폐완료
→ MA_STAT.AbCorePull_Flag = 1
→ 코어풀 조건 만족
→ 노즐터치 또는 사출 시작보다 먼저 코어풀 가능
```

이는 요구사항인 “사출과 동시에 코어풀 동작”과 맞지 않는다.

### 3.2 출력단만 수정해서는 자동 코어풀 타이밍이 바뀌지 않음

초기 접근은 `DO_5_3` 출력 조건을 조정하는 방식이었다.

하지만 실제 자동 코어풀 시작의 핵심은 `DO_5_3` 자체가 아니라 `MA_STAT.AbCorePull_Flag`이다.

`co_con.c`에서는 코어풀 옵션이 활성화된 경우 `MA_STAT.AbCorePull_Flag`를 보고 `S_CO1` 코어축을 시간동작 POS 방향으로 시작한다.

따라서 단순히 `DO_5_3` 출력단만 수정하면, 상위 코어축 상태머신의 시작 시점은 그대로 남을 수 있다.

즉, 실제 수정 지점은 출력선이 아니라 **코어풀 시작 요청 플래그의 실행 타이밍**이다.

### 3.3 노즐터치 완료만으로는 요구사항을 완전히 만족하지 않음

중간 수정안에서는 형폐완료 후 들어온 코어풀 요청을 보류하고, 노즐터치 완료 후 코어풀을 실행하도록 구성하였다.

하지만 요구사항은 단순히 “노즐터치 완료 후 코어풀”이 아니라, 다음 구조이다.

```text
형폐완료
→ 노즐터치 완료
→ 사출 시작
→ 사출과 동시에 코어풀 동작
```

따라서 노즐터치 완료 조건만으로는 부족하며, **사출 시작 조건**이 추가로 필요하다.

### 3.4 냉각수 고정측 출력 조건 부족

기존 고정측 냉각수 출력은 금형히터 ON 조건만 사용하였다.

```c
DO_WaterFlow_Fixplate = ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP];
```

이 경우 별도의 냉각수 사용 요청이 있어도 금형히터가 꺼져 있으면 고정측 냉각수 출력이 켜지지 않을 수 있다.

따라서 냉각수 사용 요청 상태와 실제 출력 상태가 불일치할 가능성이 있었다.

## 4. 주요 수정 내용

### 4.1 CorePullAutoReqHold 추가

형폐완료 후 들어온 자동/반자동 코어풀 요청을 즉시 실행하지 않고 보류하기 위해 다음 로컬 변수를 추가하였다.

```c
_LOCAL BOOL CorePullAutoReqHold;
```

이 변수는 `MA_STAT.AbCorePull_Flag`가 들어왔을 때, 이를 즉시 실행하지 않고 내부적으로 저장하는 역할을 한다.

즉, `MA_STAT.AbCorePull_Flag`는 더 이상 곧바로 코어풀 실행 신호로 사용되지 않고, 먼저 `CorePullAutoReqHold`에 의해 보류된다.

### 4.2 CorePullInjectionStartHold 추가

사출 시작 시점을 기준으로 코어풀 실행을 래치하기 위해 다음 로컬 변수를 추가하였다.

```c
_LOCAL BOOL CorePullInjectionStartHold;
```

이 변수는 노즐터치 완료 후 사출 출력이 시작되는 순간을 잡아, 코어풀 실행 시작점을 고정하는 역할을 한다.

사출 시작 후에는 순간적으로 사출 출력 신호가 흔들리더라도, 기존 `CorePullTime` 동안 코어풀 출력이 유지될 수 있도록 한다.

### 4.3 노즐터치 완료 판정 함수 추가

노즐터치 완료 조건을 명확히 하기 위해 다음 함수를 추가하였다.

```c
BOOL IsCorePullNozzleTouchEnd(void)
{
    return (AXdat[S_IU].Status == AX_STAT_NEG_END_POSITION);
}
```

현재 기준에서 노즐터치 완료는 `S_IU` 축의 `AX_STAT_NEG_END_POSITION` 상태로 판단한다.

### 4.4 사출 시작 판정 함수 추가

사출 시작 조건을 명확히 하기 위해 다음 함수를 추가하였다.

```c
BOOL IsCorePullInjectionStart(void)
{
    return (AXdat[S_IP].PhyOut.negDO1);
}
```

`tk1o.c`에서 `DO_IP_Inject`는 다음과 같이 생성된다.

```c
DO_IP_Inject = AXdat[S_IP].PhyOut.negDO1;
```

따라서 `tk2o.c`에서는 태스크 간 1스캔 지연 가능성을 줄이기 위해 `DO_IP_Inject` 미러 변수가 아니라 원천값인 `AXdat[S_IP].PhyOut.negDO1`을 직접 사용하였다.

### 4.5 코어풀 출력 허용 조건 함수 추가

코어풀 실행 인터록 조건을 함수로 분리하였다.

```c
BOOL IsCorePullOutputCondition(void)
{
    return (
        CorePullTime >= 0 &&
        !DO_CU_Differ &&
        (!DO_Remove_Runner_BWD || !DO_Remove_Runner_FWD)
    );
}
```

이 함수는 코어풀 타이머, 형체 차압 출력, 런너 제거 동작 상태를 확인한다.

노즐터치 완료와 사출 시작 조건은 별도의 단계에서 먼저 확인한 후, 사출 시작 래치 이후에 이 인터록 조건을 검사한다.

### 4.6 자동/반자동 코어풀 로직 변경

자동/반자동 코어풀 로직은 다음 순서로 변경하였다.

```text
1. 형폐완료가 아니면 코어풀 보류 상태, 사출 시작 래치, 플래그, 타이머를 초기화한다.

2. 형폐완료 후 MA_STAT.AbCorePull_Flag가 들어오면 CorePullAutoReqHold를 1로 설정한다.

3. 노즐터치 완료 전에는 MA_STAT.AbCorePull_Flag를 0으로 내려 co_con.c의 코어축 시작을 차단한다.

4. 노즐터치 완료 후에도 사출 출력이 시작되지 않았으면 MA_STAT.AbCorePull_Flag를 계속 0으로 유지한다.

5. 사출 출력 AXdat[S_IP].PhyOut.negDO1이 ON되면 CorePullInjectionStartHold를 1로 래치한다.

6. CorePullInjectionStartHold가 1이고 코어풀 출력 인터록이 만족되면 MA_STAT.AbCorePull_Flag를 1로 다시 세우고 CoreOpen()을 실행한다.

7. CorePullTime이 종료되면 MA_STAT.AbCorePull_Flag, CorePullAutoReqHold, CorePullInjectionStartHold를 모두 0으로 리셋하고 CoreClose()를 실행한다.
```

이 구조를 통해 코어풀 실행 시점은 다음과 같이 변경된다.

```text
기존:
형폐완료 후 코어풀 가능

변경:
형폐완료 후 요청 보류
→ 노즐터치 완료 확인
→ 사출 시작 확인
→ 코어풀 실행
```

### 4.7 냉각수 고정측 출력 조건 추가

기존 코드:

```c
DO_WaterFlow_Fixplate =
    ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP];
```

수정 코드:

```c
DO_WaterFlow_Fixplate =
    ACT_DATA.System.AbHeatOn[S_HEAT_MOLD_TEMP_CON_GROUP]
    || MA_STAT.AbWaterFlow_Use;
```

변경 후에는 금형히터가 켜져 있거나, 별도 냉각수 사용 요청이 있을 때 고정측 냉각수 출력이 ON된다.

```text
코어풀 요청은 보류되지만 코어풀 출력은 동작하지 않는다.
```

```text
노즐터치가 완료되어도 사출이 시작되지 않았으므로 코어풀은 동작하지 않는다.
```

## 7. 기대 효과

### 7.1 코어풀 동작 타이밍 명확화

코어풀은 더 이상 형폐완료 직후 실행되지 않는다.

또한 노즐터치 완료 직후에도 바로 실행되지 않고, 사출 출력 시작 시점에 맞춰 실행된다.

이를 통해 요구사항인 **사출과 코어풀의 동시 동작**을 만족할 수 있다.

### 7.2 기존 상태머신 영향 최소화

`co_con.c`와 `aut_main.c`는 수정하지 않았다.

따라서 기존 자동 시퀀스 계산 구조와 코어축 상태머신은 그대로 유지된다.

수정 범위는 `tk2o.c` 내부의 코어풀 실행 타이밍 제어로 제한하였다.

### 7.3 디버깅 기준 명확화

코어풀 동작 판단 기준이 다음과 같이 분리된다.

```text
요청 발생:
MA_STAT.AbCorePull_Flag

요청 보류:
CorePullAutoReqHold

노즐터치 완료:
AXdat[S_IU].Status

사출 시작:
AXdat[S_IP].PhyOut.negDO1

실행 래치:
CorePullInjectionStartHold

실제 출력:
DO_5_3
```

따라서 트렌드 확인 시 어느 단계에서 조건이 막히는지 파악하기 쉬워진다.

### 7.4 냉각수 출력 조건 확장

금형히터 ON 상태뿐 아니라 별도 냉각수 사용 요청도 출력 조건에 포함되었다.

이를 통해 고정측 냉각수 출력 조건이 실제 운전 요구사항에 맞게 확장되었다.

## 8. 확인 필요 사항

### 8.1 노즐터치 완료 상태값 확인

현재 코드는 노즐터치 완료를 다음 조건으로 판단한다.

```c
AXdat[S_IU].Status == AX_STAT_NEG_END_POSITION
```

실제 장비 트렌드에서 노즐터치 완료 시 해당 상태가 맞는지 확인해야 한다.

### 8.2 사출 시작 출력 확인

현재 코드는 사출 시작을 다음 조건으로 판단한다.

```c
AXdat[S_IP].PhyOut.negDO1
```

`tk1o.c`에서 `DO_IP_Inject`가 동일한 값으로 생성되므로 사출 출력 기준으로는 적절하다.

다만 실제 장비 트렌드에서 이 값이 사출 시작 타이밍과 일치하는지 확인해야 한다.

### 8.3 코어풀 옵션 변수 일치 확인

`tk2o.c`에서는 다음 값을 사용한다.

```c
AUT_PAR.SbCorePullEnable
```

`co_con.c`에서는 다음 값을 사용한다.

```c
REC_DATA.AUT.SbCorePullEnable
```

두 값이 실제 운전 중 동일하게 동기화되는지 확인해야 한다.

값이 다르면 `tk2o.c`와 `co_con.c`가 코어풀 옵션 상태를 다르게 판단할 수 있다.

### 8.4 MA_STAT.AbWaterFlow_Use 구조체 존재 확인

냉각수 조건에 추가한 다음 멤버가 `ma_stat_typ` 구조체에 실제로 선언되어 있어야 한다.

```c
MA_STAT.AbWaterFlow_Use
```

선언이 없으면 컴파일 에러가 발생하므로 구조체 정의 확인이 필요하다.

### 8.5 현장 모니터링 권장 변수

## 9. 결론

이번 수정의 핵심은 단순히 `DO_5_3` 출력 조건을 바꾼 것이 아니라, **코어풀 시작 명령의 실제 실행 타이밍을 사출 시작 시점으로 이동시킨 것**이다.

최종 동작 구조는 다음과 같다.

```text
형폐완료
→ aut_main.c에서 AbCorePull_Flag 요청 발생
→ tk2o.c에서 CorePullAutoReqHold로 요청 보류
→ 노즐터치 완료 확인
→ 사출 출력 AXdat[S_IP].PhyOut.negDO1 확인
→ CorePullInjectionStartHold 래치
→ MA_STAT.AbCorePull_Flag 재발생
→ co_con.c에서 S_CO1 코어축 시작
→ CoreOpen()으로 DO_5_3 출력
→ CorePullTime 종료 후 리셋
```

본 수정으로 코어풀 자동 동작은 형폐완료 기준이 아니라, **노즐터치 완료 후 사출 시작과 동시에 실행되는 구조**로 변경되었다.

또한 고정측 냉각수 출력 조건에 `MA_STAT.AbWaterFlow_Use`를 추가하여, 금형히터 ON 조건 외의 별도 냉각수 사용 요청도 반영할 수 있도록 개선하였다.
