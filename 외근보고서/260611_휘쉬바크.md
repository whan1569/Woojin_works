# 코어풀 자동 동작 현장 조치 및 1차 시도 결과 보고서

## 1. 개요

본 보고서는 외근 중 진행한 코어풀 자동 동작 관련 현장 조치 내용과, 1차 수정 시도가 기대한 결과로 이어지지 않은 원인을 정리한 것이다.

현장 요청사항은 자동/반자동 운전 중 코어풀 동작 시점을 기존보다 뒤로 조정하는 것이었다. 구체적으로는 형폐완료 이후 바로 코어풀이 동작하는 구조가 아니라, 노즐터치 완료 이후 코어풀이 동작하도록 변경하는 것이 목적이었다.

---

## 2. 현장 상황

현장에서는 장비 동작 확인과 코드 수정이 동시에 필요한 상황이었다.

당시 확인된 주요 조건은 다음과 같았다.

```text
대상 기능: 자동/반자동 코어풀 동작
대상 출력: DO_5_3
요구 동작: 노즐터치 완료 이후 코어풀 실행
작업 조건: 현장 외근 중 제한된 시간 내 원인 파악 및 1차 조치 필요
```

시간적 제약이 있었기 때문에 전체 시퀀스 구조를 모두 추적하기보다는, 현장에서 바로 확인 가능한 출력 조건 중심으로 1차 수정을 진행하였다.

---

## 3. 1차 조치 방향

1차 조치 당시에는 코어풀 실제 출력이 `DO_5_3`으로 보였기 때문에, `DO_5_3` 출력 조건에 노즐터치 완료 조건을 추가하는 방향으로 접근하였다.

당시 판단 흐름은 다음과 같았다.

```text
코어풀 출력 = DO_5_3
노즐터치 완료 후 코어풀 필요
따라서 DO_5_3 출력 조건에 노즐터치 완료 조건을 추가하면 될 것으로 판단
```

즉, 1차 조치는 코어풀의 최종 출력단을 지연시키는 방식이었다.

---

## 4. 1차 시도 결과

1차 수정 후 기대한 동작 변화가 나타나지 않았다.

현상은 다음과 같이 정리된다.

```text
수정 위치: DO_5_3 출력 조건
기대 결과: 노즐터치 완료 이후 코어풀 동작
실제 결과: 코어풀 동작 타이밍이 의도대로 변경되지 않음
```

현장에서는 출력 조건만 단순히 늦추는 방식으로는 기능 변경이 충분하지 않다는 점이 확인되었다.

---

## 5. 1차 시도가 실패한 원인

1차 시도가 실패한 이유는 수정 지점이 실제 코어풀 시작 기준과 달랐기 때문이다.

처음에는 `DO_5_3`이 코어풀 출력이므로, 해당 출력 조건을 수정하면 동작 타이밍도 변경될 것으로 판단하였다.

하지만 이후 코드 구조를 다시 확인한 결과, 코어풀 자동 동작의 시작 기준은 `DO_5_3`이 아니라 `MA_STAT.AbCorePull_Flag`였다.

실제 구조는 다음과 같이 해석된다.

```text
MA_STAT.AbCorePull_Flag
→ 코어축 시작 조건 성립
→ 코어축 상태머신 진행
→ 최종적으로 DO_5_3 출력
```

즉, `DO_5_3`은 코어풀의 최종 출력에 가깝고, 코어풀 동작 시작 여부는 상위 상태 플래그인 `MA_STAT.AbCorePull_Flag`에서 결정되고 있었다.

따라서 `DO_5_3` 출력단만 수정하면 코어축 시작 조건은 기존과 동일하게 유지된다.

---

## 6. 교착처럼 보인 이유

1차 수정 후 동작이 꼬이거나 교착처럼 보인 이유는 상태 기준과 출력 기준이 서로 달라졌기 때문이다.

1차 수정 후 구조는 다음과 같았다.

```text
상위 코어축 상태머신:
AbCorePull_Flag를 기준으로 기존 타이밍에 코어풀을 시작하려 함

출력단:
노즐터치 완료 조건이 만족되지 않아 DO_5_3 출력을 막음
```

즉, 코어축 제어부에서는 이미 코어풀 진행 조건이 성립된 것으로 판단할 수 있지만, 출력단에서는 별도 조건으로 실제 출력을 막는 구조가 되었다.

이로 인해 다음과 같은 상태 불일치가 발생할 수 있다.

```text
시퀀스는 진행하려고 함
출력은 막혀 있음
상태는 시작 조건에 걸려 있음
실제 동작은 기대대로 나오지 않음
```

이 상태가 현장에서는 교착상태처럼 보일 수 있었다.

---

## 7. DO_5_3 직접 제어에 대한 판단

초기에는 `DO_5_3`이 여러 경로에서 직접 제어되는 구조가 문제일 가능성도 고려하였다.

하지만 이후 판단으로는 `DO_5_3` 직접 제어 자체가 이번 실패의 핵심 원인은 아니었다.

기존 구조는 중간 변수 교환이 여러 번 있는 형태로 볼 수 있다.

```text
코어축 상태 또는 옵션 로직
→ 중간 명령 변수 또는 함수
→ DO_5_3 출력
```

따라서 출력 직접 제어 구조는 가독성이나 추적성 측면에서는 좋지 않지만, 기능 실패의 본질은 아니었다.

이번 실패의 본질은 최종 출력선을 수정했지만, 실제 시작 명령 플래그의 발생 타이밍은 바꾸지 못했다는 점이다.

---

## 8. 원인 정리

1차 시도 실패 원인은 다음과 같이 정리할 수 있다.

| 구분        | 1차 판단             | 실제 구조                   |
| --------- | ----------------- | ----------------------- |
| 코어풀 제어 기준 | DO_5_3 출력         | MA_STAT.AbCorePull_Flag |
| 수정 위치     | 출력단 조건            | 시작 플래그 조건이어야 함          |
| 기대 효과     | 출력 지연으로 동작 타이밍 변경 | 상위 상태머신은 기존 타이밍 유지      |
| 실패 원인     | 출력 조건만 수정         | 시작 명령 타이밍 미변경           |
| 현장 증상     | 동작 타이밍 변경 실패      | 상태와 출력 조건 불일치           |

---

## 9. 결론

외근 당시에는 시간적 제약으로 인해 코어풀 최종 출력인 `DO_5_3` 기준으로 단순 수정하는 1차 조치를 진행하였다.

그러나 1차 시도는 기대한 결과로 이어지지 않았다.

원인은 코어풀 자동 동작의 실제 시작 기준이 `DO_5_3` 출력이 아니라 `MA_STAT.AbCorePull_Flag`였기 때문이다.

따라서 출력단 조건만 변경하면 코어축 상태머신은 기존 조건대로 먼저 진행하려 하고, 출력단에서는 노즐터치 조건으로 실제 출력을 막는 상태가 되어 동작이 꼬이거나 교착처럼 보일 수 있다.

이번 건의 핵심 교훈은 다음과 같다.

```text
최종 출력 DO를 기준으로 수정하면 안 되고,
해당 출력을 발생시키는 상위 시퀀스 플래그와 상태머신 기준을 먼저 확인해야 한다.
```
