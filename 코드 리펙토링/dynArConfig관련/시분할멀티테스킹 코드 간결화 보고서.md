## **dynArConfig.c 개선 보고서**

### 1. 개요

본 문서는 `dynArConfig.c`의 구조 개선 및 리팩터링 수행 내용을 정리한 것이다.
기존 구현은 단계(step) 기반 상태 제어 구조로 작성되어 있었으며, 실제 실행 환경과 맞지 않는 설계로 인해 가독성과 유지보수성에 문제가 있었다.

---

### 2. 기존 구조의 문제점

#### 2.1 불필요한 단계 기반 상태 처리

기존 코드는 `DYN_AR.step`을 기반으로 다음과 같은 세분화된 단계로 구성되어 있었다.

* Parsing
* Node 생성
* AR 생성 / 삭제 / 저장 / 적용
* IO 생성 / 삭제 / 저장 / 적용
* NC 생성 / 삭제 / 저장 / 적용

이 구조는 **비동기 처리 환경을 가정한 설계**이나, 실제 시스템은 시분할 멀티태스킹 환경으로 동작한다.

즉,

* 작업이 중간에 외부에 의해 중단되거나
* 여러 사이클에 걸쳐 분할 수행될 필요가 없는 구조임에도

불필요하게 상태를 분리하여 처리하고 있었다.

---

#### 2.2 가독성 저하

단일 로직이 다수의 step으로 분산되어 있어 전체 흐름 파악이 어렵다.

예:

* AR 처리 로직이 여러 step에 분산
* IO / NC도 동일 패턴 반복
* 실제 수행 순서를 추적하기 위해 step 전이 흐름을 따라가야 함

이로 인해 코드 읽기 비용이 증가하며, 신규 인원이 이해하기 어려운 구조가 된다.

---

#### 2.3 유지보수성 저하

세분화된 상태 구조는 다음과 같은 문제를 유발한다.

* 로직 수정 시 여러 step을 동시에 고려해야 함
* 기능 추가 시 step 확장 필요
* 중복 패턴 (AR/IO/NC)이 반복되어 변경 영향 범위 확대
* 디버깅 시 현재 상태(step)에 의존하는 복잡한 추적 필요

결과적으로 **변경 비용 증가 및 오류 가능성 증가**로 이어진다.

---

#### 2.4 구현 디테일이 외부 상태로 노출됨

step 값은 실제 비즈니스 상태가 아니라 **내부 처리 순서를 표현하는 값**임에도,
외부에서 확인 가능한 상태로 노출되어 있다.

이는 설계적으로 다음 문제를 가진다.

* 내부 구현 변경 시 외부 영향 발생
* 상태값 자체가 의미 없는 정보가 됨
* 인터페이스와 구현이 분리되지 않음

---

### 3. 개선 방향

#### 3.1 단계 기반 상태 제거

`DYN_AR.step` 기반 상태기계를 제거하고,
전체 로직을 **단일 실행 흐름**으로 재구성하였다.

---

#### 3.2 기능 단위로 구조 재편

전체 처리를 다음과 같은 의미 단위 함수로 재구성하였다.

* `parseData()`
* `buildDynamicNodes()`
* `applyArConfig()`
* `applyIoConfig()`
* `applyNcConfig()`
* `applyDynamicConfig()`

이를 통해 **실제 처리 흐름이 코드 구조와 일치하도록 개선**하였다.

---

#### 3.3 반복 패턴 통합

AR / IO / NC 처리 구조의 공통 패턴을 정리하여 다음과 같이 통일하였다.

* 기본 데이터 로드
* 커스터마이징 적용
* 기존 데이터 삭제
* 신규 데이터 생성
* 시스템 적용

중복 구조를 동일한 흐름으로 정리하여 유지보수성을 향상시켰다.

---

#### 3.4 메모리 관리 개선

기존 코드에서는 일부 동적 메모리가 해제되지 않는 문제가 있었으며,
리팩터링을 통해 다음을 개선하였다.

* 모든 TMP_alloc 대상에 대해 명확한 해제 경로 확보
* 종료 시 일괄 정리 함수(`memoryFreeAll`) 구성

---

#### 3.5 코드 단순화 및 주석 정리

* 불필요하거나 의미 없는 주석 제거
* 단계 설명 중심 주석 삭제
* 기능 단위 중심의 상위 레벨 주석만 유지

---

### 4. 개선 결과

#### 4.1 가독성 향상

기존:

* step 흐름 추적 필요

개선:

* 함수 호출 순서만으로 전체 흐름 이해 가능

---

#### 4.2 유지보수성 향상

* 기능 단위로 코드 분리
* 수정 범위 최소화
* 반복 패턴 구조 통일

---

#### 4.3 설계 정합성 확보

* 실제 실행 환경(시분할 멀티태스킹)에 맞는 구조로 변경
* 비동기 전제 제거
* 내부 구현 상태를 외부에서 제거

---

### 5. 결론

기존 `dynArConfig.c`는 비동기 처리 구조를 가정한 단계 기반 설계로 인해
실제 시스템 환경과 맞지 않는 과도한 상태 분리를 포함하고 있었다.

본 개선을 통해:

* 불필요한 단계 처리 제거
* 코드 흐름 단순화
* 가독성 및 유지보수성 향상

을 달성하였으며,
실제 동작 구조에 부합하는 형태로 재구성하였다.
