# B&R Automation Studio 변수 구조 교육자료

## `.typ`, `.var`, GLOBAL, LOCAL, Retain 개념 정리

---

## 1. 목적

B&R Automation Studio에서 사용하는 변수 구조를 정리한다.

현장에서 자주 헷갈리는 부분은 다음과 같다.

* `.typ`와 `.var`의 차이
* GLOBAL 변수와 LOCAL 변수의 차이
* LOCAL 변수도 다른 곳에서 참조되는 이유
* 전원 재기동 후 값이 유지되는 변수의 기준
* `AsDefault.h`를 통한 실제 참조 관계 확인 방법

---

## 2. `.typ`와 `.var`의 차이

B&R 프로젝트에서는 변수 관련 파일이 보통 `.typ`, `.var`로 나뉜다.

### 2.1 `.typ`

`.typ`는 **자료형을 정의하는 파일**이다.

즉, 변수의 “모양”을 만드는 곳이다.

예시:

```pascal
TYPE
    MotorStatus : STRUCT
        Ready : BOOL;
        Error : BOOL;
        Position : REAL;
    END_STRUCT;
END_TYPE
```

위 코드는 `MotorStatus`라는 구조체 타입을 정의한 것이다.
아직 실제 변수가 만들어진 것은 아니며, 변수를 만들기 위한 틀을 정의한 것이다.

> `.typ` = 타입 정의 / 구조체 정의 / 변수의 모양

---

### 2.2 `.var`

`.var`는 **실제 변수를 선언하는 파일**이다.

예시:

```pascal
VAR_GLOBAL
    gMotor1 : MotorStatus;
    gMotor2 : MotorStatus;
END_VAR
```

위 코드는 `.typ`에서 정의한 `MotorStatus` 타입을 사용해 `gMotor1`, `gMotor2`라는 실제 변수를 선언한 것이다.

> `.var` = 실제 변수 선언 / 메모리에 올라가는 값

---

## 3. GLOBAL 변수와 LOCAL 변수

GLOBAL과 LOCAL의 차이는 단순한 메모리 차이가 아니다.

핵심은 다음과 같다.

> GLOBAL / LOCAL은 “누가 접근하느냐”, “누가 소유하느냐”의 문제이다.

---

## 4. GLOBAL 변수

GLOBAL 변수는 여러 Program, Task, Function에서 함께 사용할 목적으로 선언하는 변수이다.

예시:

```pascal
VAR_GLOBAL
    gRunMode : USINT;
    gAlarmReset : BOOL;
    gMachineReady : BOOL;
END_VAR
```

GLOBAL 변수는 장비 전체에서 공유해야 하는 값에 사용한다.

대표적인 예시는 다음과 같다.

```text
장비 운전 모드
알람 상태
HMI 설정값
장비 옵션 설정
I/O 매핑 상태
통신 상태
전체 인터록 상태
여러 프로그램이 공통으로 봐야 하는 명령값
```

GLOBAL 변수는 여러 코드에서 쉽게 공유할 수 있다는 장점이 있다.
하지만 여러 곳에서 동시에 읽고 쓰기 시작하면, 누가 언제 값을 바꿨는지 추적하기 어려워진다.

따라서 GLOBAL 변수는 **장비 전체가 공유해야 하는 값**에만 사용하는 것이 좋다.

---

## 5. LOCAL 변수

B&R에서 LOCAL 변수는 일반 C언어의 함수 내부 지역변수와 완전히 같은 개념으로 보면 안 된다.

B&R에서 LOCAL 변수는 다음 의미에 가깝다.

> 특정 Program 또는 특정 로직에 소속된 Process Variable

즉, LOCAL은 “절대 외부 접근이 불가능한 변수”라기보다, **해당 프로그램이 주인인 변수**라고 보는 것이 더 정확하다.

예시:

```pascal
VAR
    step : UINT;
    tempFlag : BOOL;
    prevInput : BOOL;
END_VAR
```

LOCAL 변수는 보통 해당 프로그램 내부에서 사용하는 값이다.

대표적인 예시는 다음과 같다.

```text
내부 Step
임시 Flag
출력 계산 중간값
Edge Detection용 이전값
해당 동작 내부 상태
한 사이클 판단 결과
외부에서 직접 건드리면 안 되는 내부 변수
```

---

## 6. LOCAL 변수도 다른 곳에서 보이는 이유

B&R에서는 LOCAL 변수라도 프로젝트 구조, include, header, PV 접근 방식에 따라 다른 코드에서 참조되는 경우가 있다.

그래서 “LOCAL인데 왜 다른 곳에서 보이지?”라는 상황이 생길 수 있다.

하지만 이때 중요한 것은 접근 가능 여부가 아니라 **소유권**이다.

> LOCAL 변수의 주인은 해당 Program이다.
> 외부에서 보일 수는 있어도, 외부에서 마음대로 쓰거나 바꾸면 안 된다.

즉, LOCAL은 공유가 안 되는 변수가 아니라, **공유하면 안 되는 책임 범위의 변수**에 가깝다.

---

## 7. GLOBAL / LOCAL 사용 기준

### GLOBAL로 두는 값

여러 프로그램이 공식적으로 공유해야 하는 값은 GLOBAL로 둔다.

```text
장비 운전 모드
알람 상태
HMI 설정값
I/O Mapping 값
장비 옵션
축 상태
히터 상태
통신 상태
공통 명령
전체 인터록
```

### LOCAL로 두는 값

특정 프로그램이 책임지고 관리해야 하는 값은 LOCAL로 둔다.

```text
해당 동작 내부 Step
임시 판단 Flag
출력 계산 중간값
Edge 검출용 이전값
한 사이클 판단값
내부 타이머 상태
특정 함수 내부에서만 쓰는 조건 결과
```

---

## 8. 위험한 변수 사용 방식

가장 위험한 패턴은 **임시 플래그를 GLOBAL처럼 사용하는 것**이다.

예시:

```text
MoveDoneFlag
TempReadyFlag
OutputRequest
동작 중 임시 완료 신호
출력 요청 임시 신호
```

이런 값이 여러 프로그램에서 동시에 읽고 쓰이기 시작하면 다음 문제가 생긴다.

```text
이전 사이클 값이 다음 동작에 섞임
누가 값을 살렸는지 추적하기 어려움
누가 값을 죽였는지 확인하기 어려움
출력 조건이 여러 코드에 흩어짐
STEP 전이가 꼬임
간헐적으로만 재현되는 오류가 발생함
현장 AS 시 원인 추적 시간이 길어짐
```

따라서 임시 플래그는 가능하면 LOCAL로 관리하고, 외부 공유가 필요한 경우에는 명확한 명령 변수와 상태 변수로 분리하는 것이 좋다.

---

## 9. 전원 재기동 후 값이 유지되는 기준

GLOBAL 변수라고 해서 전원 재기동 후 값이 유지되는 것은 아니다.
LOCAL 변수라고 해서 전원 재기동 후 값이 무조건 사라지는 것도 아니다.

값이 유지되는지는 GLOBAL/LOCAL이 아니라 다음 설정에 의해 결정된다.

```text
Retain / Remanent / Permanent 설정 여부
CPU의 RemMem / PV Memory 설정
Warm Restart / Cold Restart 여부
소프트웨어 전송 시 Keep PV Values 설정 여부
```

즉, 구분은 다음과 같다.

```text
GLOBAL / LOCAL = 접근 범위와 소유권 문제
Retain / Remanent / Permanent = 전원 재기동 후 값 유지 문제
```

---

## 10. 재기동 후 값 유지 예시

| 변수 종류     | Retain/Remanent 아님   | Retain/Remanent 설정됨 |
| --------- | -------------------- | ------------------- |
| GLOBAL 변수 | 재기동 시 초기값으로 돌아갈 수 있음 | 껐다 켜도 유지될 수 있음      |
| LOCAL 변수  | 재기동 시 초기값으로 돌아갈 수 있음 | 껐다 켜도 유지될 수 있음      |

정리하면 다음과 같다.

```text
GLOBAL + 일반 변수 = 유지 안 될 수 있음
LOCAL + 일반 변수 = 유지 안 될 수 있음
GLOBAL + Retain 변수 = 유지될 수 있음
LOCAL + Retain 변수 = 유지될 수 있음
```

---

## 11. 유지되어야 하는 변수와 유지되면 위험한 변수

### 전원 재기동 후 유지되어야 하는 변수

```text
설정값
보정값
레시피 값
누적 카운터
장비 옵션 설정
원점/캘리브레이션 관련 값
운전 파라미터
사용자가 HMI에서 저장한 값
```

### 전원 재기동 후 유지되면 위험한 변수

```text
동작 중 임시 플래그
한 사이클 판단값
자동동작 Step
출력 요청 Flag
인터록 임시 결과
Edge 검출 후 임시 상태
동작 완료 임시 신호
```

임시 플래그나 Step 값이 재기동 후에도 남아 있으면, 이전 동작의 흔적이 다음 동작에 섞일 수 있다.

따라서 임시 플래그는 Retain 대상이 되지 않도록 주의해야 한다.

## 12. `AsDefault.h`를 통한 변수 참조 확인

B&R Automation Studio의 C 코드에서는 보통 다음 헤더를 포함한다.

```c
#include <AsDefault.h>
```

`AsDefault.h`는 프로젝트에서 사용하는 기본 헤더이며, 변수 선언과 타입 정의가 코드에서 참조될 수 있도록 관련 헤더들을 포함하는 역할을 한다.

따라서 `.typ`, `.var`, GLOBAL, LOCAL 변수의 실제 참조 관계를 확인할 때는 단순히 파일명만 보고 판단하지 말고, `AsDefault.h`를 기준으로 어떤 헤더가 포함되고 있는지 확인하는 것이 좋다.

특히 LOCAL 변수라도 프로젝트 구조와 헤더 포함 방식에 따라 다른 코드에서 참조되는 것처럼 보일 수 있다.

직접적인 검증은 다음 기준으로 수행하는 것을 권장한다.

```text
1. 해당 변수가 어느 .var 파일에 선언되어 있는지 확인한다.
2. 해당 변수의 GLOBAL / LOCAL 선언 위치를 확인한다.
3. AsDefault.h 또는 관련 include 헤더를 통해 코드에서 어떻게 참조되는지 확인한다.
4. 실제 C 코드에서 해당 변수를 읽기만 하는지, 쓰기까지 하는지 확인한다.
5. 외부 프로그램에서 LOCAL 변수를 직접 수정하고 있다면 책임 범위가 흐려지는 구조로 판단한다.
```

정리하면 다음과 같다.

> `#include <AsDefault.h>`는 B&R C 코드에서 관련 변수와 타입 헤더를 참조하기 위한 기본 헤더이다.
> 따라서 GLOBAL/LOCAL 변수의 실제 접근 가능 여부는 `.var` 선언 위치뿐 아니라 `AsDefault.h` 및 관련 include 구조를 기준으로 직접 검증하는 것을 권장한다.

## 13. 실무 판단 기준

변수를 만들 때는 다음 순서로 판단한다.

```text
1. 이 값은 여러 프로그램이 같이 봐야 하는가?
   → 그렇다면 GLOBAL 후보이다.

2. 이 값의 주인은 누구인가?
   → 특정 동작 로직이 관리하는 값이면 LOCAL이 적절하다.

3. 전원 재기동 후에도 남아야 하는가?
   → 설정값, 레시피, 보정값이면 Retain/Remanent 검토가 필요하다.

4. 외부에서 값을 직접 바꿔도 안전한가?
   → 외부 수정으로 동작 순서가 꼬이면 LOCAL로 관리하거나 명령/상태 변수를 분리해야 한다.
```

## 14. 최종 요약

```text
.typ는 변수의 모양을 만드는 곳이다.
.var는 실제 변수를 선언하는 곳이다.

GLOBAL은 장비 전체가 공유하는 값이다.
LOCAL은 특정 프로그램이 소유하는 값이다.

B&R의 LOCAL은 C언어 함수 지역변수처럼 완전히 닫힌 개념은 아니다.
다만 LOCAL 변수를 외부에서 직접 수정하면 책임 범위가 흐려진다.

전원 재기동 후 값이 남는지는 GLOBAL/LOCAL이 아니라
Retain, Remanent, Permanent 설정으로 결정된다.

#include <AsDefault.h>를 통해 관련 변수와 타입 헤더가 참조되므로,
실제 접근 가능 여부는 AsDefault.h 및 include 구조를 기준으로 직접 검증하는 것이 좋다.
```

## 15. 현장 설명용 문장

GLOBAL과 LOCAL은 메모리 차이라기보다 책임 범위 차이입니다.

GLOBAL은 여러 로직이 같이 봐야 하는 장비 공용 변수이고, LOCAL은 특정 프로그램이 관리해야 하는 내부 변수입니다.

단, B&R의 LOCAL은 C언어 함수 지역변수처럼 완전히 막힌 개념은 아니므로 외부에서 보일 수 있습니다. 하지만 LOCAL 변수를 다른 프로그램에서 직접 건드리기 시작하면 책임 범위가 흐려지고 디버깅이 어려워집니다.

또한 전원 재기동 후 값이 유지되는지는 GLOBAL/LOCAL이 아니라 Retain, Remanent, Permanent 설정에 따라 결정됩니다.

C 코드에서는 `#include <AsDefault.h>`를 통해 관련 변수와 타입 헤더가 참조되므로, 실제 접근 가능 여부는 `.var` 선언 위치와 `AsDefault.h` include 구조를 함께 확인하는 것이 좋습니다.
